# Project Teardown Skill

## 🎯 为什么需要这个SKILL

我是一名研究生。和很多同学一样，因为实验室节奏、时间安排等原因，并没有真正积累起足够的实习和工程经验。如果放在前几年，我们可能还会去补开发、补后端、补项目经历，尽量证明自己“会写代码”。但在vibe coding 的时代，我越来越强烈地感觉到：**真正重要的，已经不只是“会不会写某一段代码”，而是更上层的判断力，也就是一种接近架构师的思维方式。**

比如在算法工程里，更顶层的问题往往是：

- 任务怎么定义
- 需要多少数据
- baseline 和 backbone 怎么选
- benchmark 怎么设计

而在开发和 Agent 系统里，更顶层的问题则变成了：

- 为什么这里要缓存
- 为什么这里要异步
- 为什么数据要分层存储
- 为什么这里需要中间件
- 为什么一个 RAG / memory 方案会被逼出来

对没有工程经验的人来说，如果现在从头开始一门门补 Redis、中间件、并发、异步、缓存一致性这些知识，当然也能学。但问题在于：**效率太慢，而且很容易停留在“知道概念”，却不知道这些技术为什么会在真实项目里被逼出来。**并且现在代码确实已经越来越廉价了。

所以我最近在尝试一种新的学习方式：

> **从成熟项目入手，反向学习架构。**



这个 SKILL 想解决的不是“项目分析不够全”，而是另一个问题：

> **对于工程经验不足的人来说，怎么从一个项目里抓住真正重要的东西。**

我希望它做到的事情是：

- 不只告诉我“用了什么技术”
- 更要告诉我“为什么要这样设计”
- 不这样做会出什么问题
- 当前方案解决了什么
- 带来了什么代价
- 将来什么条件下要升级

在项目拆解的过程中，我能把握到项目的业务场景、技术壁垒等等。
