# 项目总览

> Skill usage marker: used the project-teardown skill

## 这是什么项目

这是一个仿大众点评的课程型项目，业务核心是四件事：

- 用户用短信验证码登录
- 查询店铺与店铺分类
- 发布探店笔记、点赞、关注、收件箱推送
- 抢购秒杀优惠券并生成订单

仓库里实际有两个后端：

- `hm-dianping`：偏教学骨架，很多接口还是 `TODO`
- `NoteFromVisit`：更完整的实现，包含登录拦截、缓存、Redis Geo、秒杀、MQ 等关键机制

如果要分析“这个项目是怎么跑起来的”，应该以 `NoteFromVisit` 为主，把 `hm-dianping` 看成早期模板或课程初始代码。

## 它解决什么业务痛点

这个项目不是在做复杂领域建模，而是在用一个很容易理解的本地生活场景，承载几类典型后端问题：

- 热点数据读多写少，数据库扛不住，需要缓存
- 登录态不能只靠单机 Session，需要可共享的会话状态
- 附近商户查询不能只靠普通数据库分页，需要地理位置索引
- 秒杀下单会瞬时打爆数据库，需要先在 Redis 做原子判断，再异步落库
- 社交流不是简单分页，而是“关注谁就收谁的动态”，需要推送式收件箱模型

所以它的价值不在于“点评业务本身多复杂”，而在于它把缓存、高并发、登录态、社交 feed 这些工程问题装进了一个容易理解的产品壳里。

## 一条核心请求是怎么流动的

以“用户抢秒杀券”为例，链路大致是：

1. 前端页面由 `nginx-1.18.0/html/hmdp` 提供静态资源。
2. 浏览器调用 Spring Boot 接口，例如 `/voucher-order/seckill/{id}`。
3. `RefreshInterceptor` 从请求头取 `authorization`，去 Redis 查登录用户，把用户信息放进 `ThreadLocal`。
4. `VoucherOrderServiceImpl.seckillVoucher()` 先在 Redis 中执行 Lua 脚本，原子判断库存是否够、用户是否重复下单。
5. Lua 判断通过后，服务端先生成订单号，再把订单消息投递到 RabbitMQ。
6. `seckillVoucherListener` 消费 MQ 消息，调用 `handleVoucherOrder()`。
7. 订单处理阶段再用 Redisson 按用户加锁，防止并发重复建单。
8. 最后在事务里扣减数据库库存并写入订单表。

这条链路体现了项目最核心的设计思想：**把高并发判定前移到 Redis，把慢而强一致的写库动作后移到异步消费者。**

## 这个项目最核心的架构思想

它本质上不是微服务，而是一个“Redis 很重”的单体应用。

单体的好处是：

- 业务流程集中，适合教学和快速验证
- 不需要引入服务拆分、注册发现、分布式事务这些额外复杂度

Redis 被赋予了多重职责：

- 登录态存储
- 验证码存储
- 热点店铺缓存
- GEO 附近商铺索引
- 博客点赞集合
- 关注集合
- 粉丝收件箱 feed
- 用户签到 bitmap
- 秒杀原子校验
- 分布式锁

这说明项目真正想训练的不是“DDD 建模”，而是“在单体里如何用 Redis 撑起高并发和高读流量场景”。

## 当前最值得优先理解的 4 个机制

### 1. 登录态为什么放 Redis，而不是只用 Session

因为前端请求需要一个跨请求、可刷新、可扩展的用户态容器。项目用 Redis Hash 存用户简要信息，用拦截器做自动续期，再用 `ThreadLocal` 给业务层取当前用户。

### 2. 店铺缓存为什么要做穿透、击穿、逻辑过期

因为店铺详情是最典型的热点读场景。如果只做简单缓存，热点失效时会直接把压力打回 MySQL。这里专门把“空值缓存 + 逻辑过期 + 后台重建”作为一个独立问题来处理。

### 3. 秒杀为什么不能直接同步写库

因为真正的瓶颈不是接口代码，而是高并发下的库存判断和订单写入。项目把“资格判断”放在 Redis Lua 中原子执行，再异步入库，避免数据库成为流量入口。

### 4. 关注推送为什么不用普通分页

因为“我关注的人发了什么”天然更适合收件箱模型。项目把博客 ID 直接推到粉丝的 ZSet 中，再按时间戳做滚动分页，这和传统的按表分页是两种完全不同的思路。

## 这个方案为什么适合当前阶段

它选择的是一个很典型的“够用而不重”的中间形态：

- 比纯数据库 CRUD 更强，能处理热点、并发和社交流
- 比真正大型分布式系统更轻，不需要拆很多服务

也就是说，它故意停在一个适合学习工程权衡的位置：

- 用了 Redis、MQ、分布式锁，已经能体现复杂系统的关键矛盾
- 但整体还是单体，便于理解全链路，不会被基础设施吞掉重点

## 目前能确认的边界

确认内容：

- 完整实现主要在 `NoteFromVisit`
- 架构是 Spring Boot 单体 + MySQL + Redis + RabbitMQ + Nginx 静态前端
- Redis 是系统的核心基础设施，不只是缓存

合理推断：

- 这是一个带明显课程/训练目的的项目，重点在工程机制演示，不是生产级闭环
- 某些设计是为讲解思路而写，未必是生产最优实现

当前未知或不完整：

- 自动化测试覆盖很少
- 生产部署、监控、告警、灰度、限流基本没有体现
- `hm-dianping` 与 `NoteFromVisit` 的关系没有在仓库中正式说明

原因是这个问题一旦理解透，后面这些机制都会串起来：

- 登录态为什么不放 Session
- 秒杀为什么先走 Lua 再走 MQ
- Feed 为什么要用 ZSet
- 签到为什么用 Bitmap
- 附近商铺为什么用 GEO

如果你愿意，我下一轮可以只深入一个主题：

## 深挖一：为什么 Redis 必须从“缓存”升级成“核心状态层”

这一轮只解释一个核心问题：为什么这个项目里的 Redis 不能只当缓存。

如果 Redis 只是缓存，那么它的职责应该是：

- MySQL 里有真数据
- Redis 只是副本
- 缓存没了，最多是请求变慢

但这个项目里的很多关键链路不是“快读”问题，而是“高并发下先判断、先协调、先隔离”的问题。只靠数据库扛这些运行时状态，系统会很快暴露出瓶颈。

### 1. 登录态不是缓存出来的，而是在线会话状态

验证码和登录 token 都是短生命周期、跨请求共享、需要续期的状态。

项目里：

- 验证码写在 Redis
- 用户登录后把简化用户信息写进 Redis Hash
- 每次请求由拦截器从 Redis 恢复用户并续期

这说明 Redis 在这里承接的是“在线用户会话状态”，不是“把用户表缓存一下”。

如果这些信息只放数据库：

- 每次请求都要查数据库恢复用户态
- 续期动作也会持续打数据库
- 数据库会被迫承担会话中心角色

这既重，也不符合关系型数据库的强项。

### 2. 秒杀资格判断必须在数据库之前完成

秒杀链路的关键不是“把优惠券信息缓存起来”，而是：

- 库存是否够
- 用户是否已经下过单
- 这些判断要原子完成
- 大量非法请求要在进入数据库前被挡住

项目在 Redis 中通过 Lua 先做库存和一人一单判断，成功后才继续走 MQ 和事务落库。

这意味着 Redis 在这里承担的是“并发闸门”和“预判层”。

如果这一步改成完全由 MySQL 承担：

- 秒杀流量会直接冲数据库
- 连接池、行锁、事务冲突会先成为瓶颈
- 后面的异步削峰价值会被大幅削弱

所以这里 Redis 不是在缓存数据库结果，而是在替数据库拦截高并发流量。

### 3. 社交关系和 Feed 更像在线状态，而不是查询结果缓存

项目把：

- 关注关系放进 Redis Set
- 点赞用户放进 Redis ZSet
- 粉丝收件箱放进 Redis ZSet

原因不是为了少查几次数据库，而是这些结构本身就适合：

- 判断是否关注 / 是否点赞
- 求共同关注
- 按时间顺序分页拉取 feed

这类数据一旦完全压回 MySQL，就会变成大量高频查询、排序和集合运算。

所以 Redis 在这里维护的是“关系状态”和“内容分发状态”，不是普通缓存副本。

### 4. 只有一部分 Redis 用法，才是传统意义上的缓存

这个项目里真正接近传统缓存概念的，主要是店铺详情、店铺分类、附近商户这些读多写少的数据。

但即便是店铺详情，项目也已经不满足于“查不到就回源”这么简单，而是继续引入了：

- 空值缓存
- 逻辑过期
- 后台重建

这说明系统已经进入一个阶段：Redis 不再只是数据库前面的加速层，而是系统运行机制的一部分。

### 结论

这个项目把 Redis 升级成核心状态层，本质上是在做一件事：

> 让 MySQL 负责最终事实，让 Redis 负责运行时状态。

MySQL 适合：

- 订单
- 用户
- 店铺
- 优惠券

Redis 适合：

- 会话状态
- 秒杀资格状态
- 点赞 / 关注关系状态
- Feed 收件箱状态
- 热点读状态

这不是技术堆砌，而是业务约束逼出来的分工。

如果不这么做，数据库就必须同时承担：

- 最终数据存储
- 在线会话中心
- 高并发判定入口
- 社交关系实时计算

这会让一个本来还能跑得动的单体，很快被运行时压力拖垮。

### 这一轮最该记住的一句话

这个项目真正的架构变化，不是“用了 Redis”，而是：

**它把 Redis 从数据库前面的缓存，提升成了系统运行时状态的承接层。**

## 深挖二：为什么这个项目在秒杀场景里要引入消息队列

这一轮只讲一个问题：为什么这个项目需要 MQ，而不是让秒杀请求直接同步写订单表。

先说结论：

> 这里引入消息队列，不是为了“系统显得高级”，而是为了把秒杀链路切成两段：
> 
> - 前半段只负责高并发资格判断
> - 后半段才负责慢得多的数据库写入

### 这个问题是怎么被逼出来的

秒杀接口面对的不是普通下单流量，而是瞬时并发。

如果请求一进来就直接做这些事：

- 查库存
- 判断一人一单
- 扣减库存
- 写订单表

那么数据库就会直接暴露在流量洪峰前面。

即便你已经用了 Redis Lua 做资格判断，只要后面还是同步写库，请求线程依然会被数据库操作占住。流量一大，接口线程、连接池、事务冲突、数据库行锁都会开始堆积。

所以真正的问题不是“能不能写成功”，而是：

**高并发资格判断和数据库持久化，根本不应该放在同一个同步请求里完成。**

### 这个项目是怎么拆这条链路的

秒杀主流程在 `VoucherOrderServiceImpl.seckillVoucher()` 中：

1. 先取当前用户
2. 先生成订单号
3. 在 Redis 中执行 Lua 脚本，原子判断库存和一人一单
4. Lua 判断通过后，不直接落库，而是把 `VoucherOrder` 投递到 RabbitMQ
5. 接口立即返回订单号

消费端由监听器接手：

1. RabbitMQ 把消息投递给 `seckill.order.queue`
2. `seckillVoucherListener` 监听这个队列
3. 监听器调用 `handleVoucherOrder()`
4. 再通过 Redisson 锁 + 事务方法 `createVoucherOrder()` 真正扣库存并写订单

也就是说，这里 MQ 的职责不是业务编排，而是：

**把“抢资格”和“写订单”之间加一个缓冲带。**

### 为什么 Redis 还不够，必须再加 MQ

很多人第一次看到这类项目会问：

“前面不是已经有 Redis Lua 了吗，为什么还要 MQ？”

因为 Lua 解决的是：

- 原子判断
- 快速拦截非法请求

但 Lua 不解决：

- 数据库写入耗时
- 请求线程阻塞
- 突发流量对数据库的直接冲击

如果没有 MQ，那么秒杀接口虽然已经把大量非法请求挡在 Redis 外面，但所有成功请求还是会立刻同步打向数据库。流量峰值一高，数据库仍然可能成为瓶颈。

MQ 在这里承担的是“削峰填谷”：

- 请求高峰时先把订单消息积压在队列里
- 消费端按自己的能力稳定落库
- 数据库面对的是被平滑后的写入流量，而不是瞬时洪峰

这一步的价值，不在于异步本身，而在于：

**数据库终于不再直接面对秒杀入口流量。**

### 为什么这个项目只把 MQ 用在秒杀，而不是到处都上

当前仓库里，MQ 主要只服务一个业务场景：秒杀订单异步落库。

这其实是合理的。

因为不是所有业务都值得引入异步消息：

- 普通店铺查询是读请求，不需要 MQ
- 登录态维护更适合 Redis，不需要 MQ
- 点赞、关注这类操作当前量级下同步处理也还能接受

只有秒杀下单具备三个同时成立的特征：

- 瞬时并发极高
- 后端落库成本明显更高
- 用户又不要求“接口返回前必须写库完成”

只有在这种场景下，MQ 的工程收益才明显大于复杂度成本。

### 这个项目里还能看到什么架构演进痕迹

`VoucherOrderServiceImpl` 里保留了两套更早的异步方案痕迹：

- 本地阻塞队列
- Redis Stream

最后切到 RabbitMQ，说明作者其实在逐步演进：

- 先证明“异步落库”这件事是必要的
- 再从进程内队列走向可独立承接消息的中间件

这很像真实项目的演化路径，而不是一步到位设计出来的。

### 当前方案解决了什么，也牺牲了什么

它解决了：

- 秒杀请求不必同步等数据库
- 数据库写入流量被削峰
- 下单资格判断和持久化解耦
- 消费端可以单独控制处理节奏

它也引入了新的问题：

- 下单链路从同步变成最终一致
- 消息可靠性、重复消费、失败重试要额外考虑
- 监控和排障比同步接口更复杂

不过对这个项目当前阶段来说，这个取舍是合理的。

因为它最想讲清楚的，不是完整的消息治理体系，而是一个核心工程直觉：

**当入口请求非常快、后端持久化非常慢时，中间应该有一个异步缓冲层。**

### 这一轮最该记住的一句话

这个项目里的 MQ，不是“通知工具”，而是：

**秒杀链路里把高并发入口和数据库写入隔开的削峰缓冲层。**

## 深挖三：为什么店铺缓存最终选择“逻辑过期 + 后台重建”

这一轮只讲缓存设计里最核心的一个选择：

> 为什么这个项目没有停留在“缓存 TTL 到期后回源”，也没有把“互斥锁回源”作为主方案，
> 而是最终在店铺详情上启用了“逻辑过期 + 后台重建”。

### 这个问题真正要解决的，不是缓存命中率，而是热点 key 失效时数据库会不会被打穿

店铺详情接口是高频读接口。

如果只是最普通的 cache aside：

- Redis 有就返回
- Redis 没有就查 MySQL
- 再把结果写回 Redis

平时当然能工作。

但一旦热点店铺 key 正好失效，短时间内大量请求会一起落到数据库。此时真正危险的不是“慢一点”，而是：

**Redis 从流量护城河瞬间失效，MySQL 被迫直面热点洪峰。**

所以缓存设计在这里的主问题其实是：

**热点 key 过期时，系统要优先保证可用性，还是优先保证绝对最新。**

### 方案一：简单 TTL，为什么不够

简单 TTL 的好处很直接：

- 实现最简单
- 数据会自动过期
- 不需要额外线程和锁

但它有一个致命问题：

- key 一旦过期，缓存中就真的什么都没有了

这意味着所有并发请求都会把数据库当作兜底入口。

对于普通冷数据，这不算严重。
但对于店铺详情这类热点数据，这相当于把数据库放到了用户请求最前线。

所以简单 TTL 只适合：

- 访问量不高
- 单 key 热度不集中
- 回源代价不大

而这里的店铺详情显然不是这种场景。

### 方案二：互斥锁回源，为什么还不够理想

项目里其实保留了互斥锁版本 `queryWithMutex()`，但没有作为最终主方案。

它的思路是：

- key 失效后，先抢锁
- 只有一个线程回源数据库
- 其他线程等待，稍后重试

这个方案已经比简单 TTL 强很多，因为它避免了数据库被所有线程同时打穿。

但它的问题也很明显：

- 用户请求会被阻塞
- 热点 key 一旦失效，很多线程会进入等待或重试
- 延迟会集中抖动

也就是说，互斥锁方案主要保护了数据库，但没有很好保护用户请求延迟。

它更像是在说：

> “数据库别死，前端慢一点可以接受。”

对于某些后台管理类接口，这没问题。
但店铺详情是前台高频读接口，用户体验同样重要。

### 方案三：逻辑过期，核心不是“过期”，而是“过期后仍然可读”

项目最终启用的是 `queryWithLogicalExpire()`。

它和简单 TTL 最大的区别是：

- Redis 里的 key 不靠真实过期时间直接删掉
- value 内部自己带一个逻辑过期时间

请求命中缓存后：

- 如果逻辑时间还没过，就直接返回
- 如果逻辑时间过了，也先返回旧值
- 同时尝试加锁，由一个后台线程去重建缓存

这件事的本质非常关键：

**过期不再等于不可用。**

这意味着热点 key 就算过了逻辑时间，用户请求依然不会集体打到数据库，因为缓存里始终还有一份可读的旧数据。

于是系统完成了一个很重要的取舍：

- 牺牲一点短时间的数据新鲜度
- 换取热点场景下的稳定低延迟和数据库安全

### 为什么这个取舍在这里是合理的

因为“店铺详情”天然是一个可以容忍轻微陈旧的读场景。

用户查看一个店铺时：

- 名称晚几秒更新，通常能接受
- 地址、图片、评分不是秒级强一致数据
- 比起绝对最新，更重要的是请求稳定返回

这就是为什么逻辑过期特别适合这里。

如果换成库存、余额、订单状态，就不能这么做。
因为那类数据的核心要求是正确性，不是高可用读。

所以不是“逻辑过期更高级”，而是：

**它更适合这种允许短暂陈旧、但不能接受热点抖动的场景。**

### 这个方案真正聪明的地方是什么

不是用了线程池，也不是用了锁。

真正聪明的是它把问题改写了：

原本的问题是：

- 缓存过期了怎么办

它改写成：

- 缓存过期后，能不能先继续服务，再慢慢修

这就是典型的系统设计思路变化：

- 不是追求每次都拿最新值
- 而是优先保证系统在高峰时仍然稳住

### 这个项目当前缓存设计的边界

虽然方向是对的，但实现仍然偏教学型。

比如：

- 锁的 TTL 很短
- 重建线程提交后就释放锁
- 没有专门的热点 key 识别和预热体系
- 没有看到完整的缓存监控与补偿机制

所以你应该学的是这套方案背后的判断逻辑，而不是把当前实现细节原样当成生产模板。

### 这一轮最该记住的一句话

这个项目选择逻辑过期，不是为了“让缓存更复杂”，而是因为：

**对于热点店铺详情，系统宁可短暂返回旧值，也不能在 key 失效时让数据库暴露在并发洪峰前面。**

## 深挖四：缓存穿透、击穿、雪崩在这个项目里分别是什么

这一轮只解决一个常见混淆：

> 缓存穿透、缓存击穿、缓存雪崩听起来都像“缓存失效了”，
> 但它们保护的根本不是同一件事。

如果这三个概念混在一起学，最后只会记成三句口号。
真正有价值的是搞清楚：

- 各自是什么流量形态
- 各自打穿的是哪一层
- 为什么这个项目对应采用了不同策略

### 1. 缓存穿透：查的是根本不存在的数据

穿透的关键词不是“热点”，而是：

**请求的目标在数据库里本来就不存在。**

例如：

- 用户传了一个根本不存在的店铺 ID
- 恶意请求不断构造不存在的 key

这时如果系统逻辑是：

- Redis 没有
- 去查 MySQL
- MySQL 也没有
- 下次还继续查

那么这些“无效请求”会稳定绕过缓存，直接打数据库。

这个项目处理穿透的方式很明确：

- 如果数据库查不到，就把空值写进 Redis
- 并且给这个空值一个短 TTL

也就是说，它承认：

**“不存在”本身也是一种可以缓存的结果。**

在代码里，空值缓存逻辑出现在 `queryWithPassThrough()` 中。

所以穿透主要保护的是：

- 数据库不要反复为“并不存在的数据”做无意义证明

### 2. 缓存击穿：查的是热点 key，而且刚好失效

击穿的关键词不是“不存在”，而是：

**一个真实存在、并且访问量很高的 key，在某个时刻失效了。**

例如：

- 店铺 1 是热点店铺
- 大量用户都在查它
- 结果这个 key 正好过期

这时问题就不是无效请求，而是大量真实请求在同一时刻一起回源。

所以击穿和穿透的根本区别是：

- 穿透：请求对象不存在
- 击穿：请求对象存在，而且很热

这个项目主要用两种思路处理击穿：

- 互斥锁回源
- 逻辑过期 + 后台重建

最终主方案选的是后者。

因为它真正想保护的是：

- 热点 key 失效时数据库不要被打爆
- 同时用户请求也尽量不要被大面积阻塞

所以击穿主要保护的是：

- 热点流量高峰下，数据库和接口延迟都要稳住

### 3. 缓存雪崩：不是一个 key 出问题，而是一批 key 一起出问题

雪崩和击穿最容易混。

击穿通常是：

- 一个热点 key 失效

雪崩通常是：

- 一大批 key 在同一时间集中失效
- 或者 Redis 整体不可用

比如：

- 很多缓存预热时用了完全相同的 TTL
- 到点后成片过期

或者：

- Redis 整体故障，缓存层整体失守

这时数据库面对的不是单个热点的回源流量，而是整个系统读流量的大面积回灌。

所以雪崩保护的不是某个 key，而是：

**整个缓存层不要在同一时间大面积失守。**

这个项目里虽然没有做完整的生产级雪崩治理，但能看到两个明显意识：

- 在部分方案里给 TTL 加随机值，避免同批 key 同时过期
- 对热点 key 更倾向于逻辑过期，而不是依赖真实 TTL 同时失效

这说明作者知道雪崩问题的本质是“过期时间同质化”。

### 4. 为什么这三个问题不能用同一种手段解决

因为它们打来的流量，性质根本不同。

穿透面对的是：

- 不存在的数据
- 重点是挡住无意义回源

击穿面对的是：

- 存在而且很热的数据
- 重点是热点 key 失效时保护数据库

雪崩面对的是：

- 多 key 或整个缓存层同时失守
- 重点是系统级流量回灌

所以策略自然不同：

- 穿透：空值缓存、布隆过滤器
- 击穿：互斥锁、逻辑过期、热点预热
- 雪崩：随机 TTL、多级缓存、限流降级、缓存高可用

如果把三者都简化成“缓存失效”，你就会误以为一个空值缓存能解决所有问题，这显然不对。

### 5. 在这个项目里，你应该怎么对应记忆

用“店铺详情”这个场景去记最清楚：

- 查一个根本不存在的店铺 ID  
  这是穿透

- 查一个大家都在访问的热门店铺，key 刚好过期  
  这是击穿

- 大量店铺缓存因为 TTL 设计不合理，在同一时刻一起过期  
  这是雪崩

一旦你能把这三个业务画面区分开，后面的技术策略就不容易混。

### 这一轮最该记住的一句话

这三个概念的区别，不在于“缓存有没有失效”，而在于：

**到底是不存在的数据绕过缓存、单个热点 key 失守，还是整层缓存同时失守。**

## 深挖五：这个项目里的社交流到底在解决什么问题

这一轮只讲社交流的一个核心问题：

> 为什么这里不能只靠数据库把博客、关注、点赞查出来，
> 而要专门引入 Set、ZSet 和收件箱 Feed 这套设计。

如果不先把这个问题想清楚，看到代码时很容易误以为：

- 点赞只是加一减一
- 关注只是多一张关系表
- Feed 只是多一个查询接口

但这三个能力放在一起后，系统面对的已经不是简单 CRUD，而是：

**“一个人发内容以后，怎么让关注他的人高效看到；一个人又怎么快速判断自己和内容的关系。”**

### 1. 社交流的核心，不是“存内容”，而是“维护人与内容的关系”

在店铺点评业务里，博客内容本身当然可以放在数据库里。

真正难的不是存博客，而是这些实时关系：

- 我有没有点赞这篇博客
- 哪些人最近点过赞
- 我关注了谁
- 我和另一个用户的共同关注是谁
- 我关注的人刚发了什么，我怎么按时间往下刷

这些问题一旦多起来，数据库表虽然也能做，但代价会迅速上升：

- 高频查询
- 集合运算
- 时间排序
- 滚动分页

所以这个项目在社交流上真正做的事，不是“把博客放进 Redis”，而是：

**把人与内容、人与人之间的在线关系状态放进 Redis。**

### 2. 点赞为什么用 ZSet，而不是只改数据库字段

博客点赞在数据库里当然也有 `liked` 计数，但这还不够。

系统还要回答两个实时问题：

- 当前登录用户是否点过赞
- 最近点赞的前几个人是谁

如果只靠数据库计数字段，你只能知道总数，不知道是哪些人点的，更不知道顺序。

所以这里额外用 `blog:liked:{blogId}` 这个 ZSet：

- member 是用户 ID
- score 是点赞时间戳

这样就同时获得了三种能力：

- 判断是否点赞：查 score 是否存在
- 取消点赞：删除 member
- 查最近点赞用户：按分数顺序取前几名

也就是说，点赞在这里不是“给博客表 liked + 1”这么简单，
而是维护一份“内容和用户之间的实时关系索引”。

### 3. 关注为什么用 Set，而不是每次都去关系表算

关注关系在数据库里也会落表，但项目同时把它放进了 Redis Set。

原因不是为了少查一条 SQL，而是因为社交关系很快就会出现集合问题：

- 我是否关注了某人
- 我和你共同关注了谁

尤其是共同关注，这本质上就是集合交集。

Redis Set 对这种问题天然合适：

- follow A：往集合里加一个用户 ID
- unfollow A：从集合里删掉
- common follow：直接做交集运算

这里你应该看到的不是“Redis 很快”，而是：

**业务问题本身就是集合问题，所以它需要集合型数据结构。**

### 4. Feed 为什么不用普通分页，而要做“收件箱推送”

这部分最值得学。

很多人第一反应是：

- 我关注了很多人
- 直接查“我关注的人发的博客”
- 按时间排序分页不就行了

这个思路在关注量和内容量很小时可以工作。
但一旦数据量上来，它会变成一个典型的高成本查询：

- 先查关注列表
- 再查这些人的内容
- 再按时间排序
- 再做分页

而这个项目换了思路：

- 用户发博客时，不是等粉丝来查
- 而是主动把博客 ID 推到每个粉丝的收件箱里

这个收件箱就是 `feed:{userId}` 对应的 ZSet：

- member 是 blogId
- score 是发布时间戳

这样粉丝拉 Feed 时，不是做全局查询，而是直接从自己的收件箱里读。

这就是典型的“推模式”收件箱，而不是“拉模式”聚合查询。

它解决的核心问题是：

**把“读取时做大查询”改成“写入时做关系分发”。**

### 5. 为什么 Feed 要用滚动分页，而不是普通 pageNum

Feed 是按时间流动的，用户不断下拉，期间还可能有新内容插入。

如果这里用传统的页码分页：

- 第 1 页、第 2 页、第 3 页

一旦中间插入新数据，后面的页码就会漂移：

- 可能重复
- 可能漏数据

所以这里改成按 `lastId + offset` 的滚动分页。

本质上，它不是在说“第几页”，而是在说：

- 上一次我读到哪个时间点
- 这个时间点上已经消费了几条

这正是时间流式内容更适合的分页方式。

所以社交流这一块真正体现出的不是 Redis 技巧，而是：

**分页模型必须跟数据流动方式匹配。**

### 6. 这套社交流方案为什么适合当前阶段

它的优点是非常明确的：

- 点赞、关注、共同关注都能快速响应
- Feed 拉取不依赖复杂聚合查询
- 业务链路清楚，适合单体项目承载

它的代价也同样明确：

- 发一篇博客时要给所有粉丝做一次写扩散
- 用户粉丝量很大时，写入成本会上升
- 收件箱数据会膨胀，需要后续考虑裁剪和归档

所以这套方案不是“永远最优”，而是：

**在当前阶段，用写扩散换读性能和实现简单度。**

这很适合课程型项目，也适合中小规模社交流。

### 这一轮最该记住的一句话

这个项目的社交流设计，本质上不是“博客加点赞和关注”，而是：

**用 Redis 维护人与内容、人与人之间的实时关系，并把首页 Feed 从查询问题改写成收件箱读取问题。**
