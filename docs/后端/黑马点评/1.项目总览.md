# 项目总览

> Skill usage marker: used the project-teardown skill

## 这是什么项目

这是一个仿大众点评的课程型项目，业务核心是四件事：

- 用户用短信验证码登录
- 查询店铺与店铺分类
- 发布探店笔记、点赞、关注、收件箱推送
- 抢购秒杀优惠券并生成订单

仓库里实际有两个后端：

- `hm-dianping`：偏教学骨架，很多接口还是 `TODO`
- `NoteFromVisit`：更完整的实现，包含登录拦截、缓存、Redis Geo、秒杀、MQ 等关键机制

如果要分析“这个项目是怎么跑起来的”，应该以 `NoteFromVisit` 为主，把 `hm-dianping` 看成早期模板或课程初始代码。

## 它解决什么业务痛点

这个项目不是在做复杂领域建模，而是在用一个很容易理解的本地生活场景，承载几类典型后端问题：

- 热点数据读多写少，数据库扛不住，需要缓存
- 登录态不能只靠单机 Session，需要可共享的会话状态
- 附近商户查询不能只靠普通数据库分页，需要地理位置索引
- 秒杀下单会瞬时打爆数据库，需要先在 Redis 做原子判断，再异步落库
- 社交流不是简单分页，而是“关注谁就收谁的动态”，需要推送式收件箱模型

所以它的价值不在于“点评业务本身多复杂”，而在于它把缓存、高并发、登录态、社交 feed 这些工程问题装进了一个容易理解的产品壳里。

## 一条核心请求是怎么流动的

以“用户抢秒杀券”为例，链路大致是：

1. 前端页面由 `nginx-1.18.0/html/hmdp` 提供静态资源。
2. 浏览器调用 Spring Boot 接口，例如 `/voucher-order/seckill/{id}`。
3. `RefreshInterceptor` 从请求头取 `authorization`，去 Redis 查登录用户，把用户信息放进 `ThreadLocal`。
4. `VoucherOrderServiceImpl.seckillVoucher()` 先在 Redis 中执行 Lua 脚本，原子判断库存是否够、用户是否重复下单。
5. Lua 判断通过后，服务端先生成订单号，再把订单消息投递到 RabbitMQ。
6. `seckillVoucherListener` 消费 MQ 消息，调用 `handleVoucherOrder()`。
7. 订单处理阶段再用 Redisson 按用户加锁，防止并发重复建单。
8. 最后在事务里扣减数据库库存并写入订单表。

这条链路体现了项目最核心的设计思想：**把高并发判定前移到 Redis，把慢而强一致的写库动作后移到异步消费者。**

## 这个项目最核心的架构思想

它本质上不是微服务，而是一个“Redis 很重”的单体应用。

单体的好处是：

- 业务流程集中，适合教学和快速验证
- 不需要引入服务拆分、注册发现、分布式事务这些额外复杂度

Redis 被赋予了多重职责：

- 登录态存储
- 验证码存储
- 热点店铺缓存
- GEO 附近商铺索引
- 博客点赞集合
- 关注集合
- 粉丝收件箱 feed
- 用户签到 bitmap
- 秒杀原子校验
- 分布式锁

这说明项目真正想训练的不是“DDD 建模”，而是“在单体里如何用 Redis 撑起高并发和高读流量场景”。

## 当前最值得优先理解的 4 个机制

### 1. 登录态为什么放 Redis，而不是只用 Session

因为前端请求需要一个跨请求、可刷新、可扩展的用户态容器。项目用 Redis Hash 存用户简要信息，用拦截器做自动续期，再用 `ThreadLocal` 给业务层取当前用户。

### 2. 店铺缓存为什么要做穿透、击穿、逻辑过期

因为店铺详情是最典型的热点读场景。如果只做简单缓存，热点失效时会直接把压力打回 MySQL。这里专门把“空值缓存 + 逻辑过期 + 后台重建”作为一个独立问题来处理。

### 3. 秒杀为什么不能直接同步写库

因为真正的瓶颈不是接口代码，而是高并发下的库存判断和订单写入。项目把“资格判断”放在 Redis Lua 中原子执行，再异步入库，避免数据库成为流量入口。

### 4. 关注推送为什么不用普通分页

因为“我关注的人发了什么”天然更适合收件箱模型。项目把博客 ID 直接推到粉丝的 ZSet 中，再按时间戳做滚动分页，这和传统的按表分页是两种完全不同的思路。

## 这个方案为什么适合当前阶段

它选择的是一个很典型的“够用而不重”的中间形态：

- 比纯数据库 CRUD 更强，能处理热点、并发和社交流
- 比真正大型分布式系统更轻，不需要拆很多服务

也就是说，它故意停在一个适合学习工程权衡的位置：

- 用了 Redis、MQ、分布式锁，已经能体现复杂系统的关键矛盾
- 但整体还是单体，便于理解全链路，不会被基础设施吞掉重点

## 目前能确认的边界

确认内容：

- 完整实现主要在 `NoteFromVisit`
- 架构是 Spring Boot 单体 + MySQL + Redis + RabbitMQ + Nginx 静态前端
- Redis 是系统的核心基础设施，不只是缓存

合理推断：

- 这是一个带明显课程/训练目的的项目，重点在工程机制演示，不是生产级闭环
- 某些设计是为讲解思路而写，未必是生产最优实现

当前未知或不完整：

- 自动化测试覆盖很少
- 生产部署、监控、告警、灰度、限流基本没有体现
- `hm-dianping` 与 `NoteFromVisit` 的关系没有在仓库中正式说明
