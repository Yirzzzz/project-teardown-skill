这一轮只解决一个常见混淆：

> 缓存穿透、缓存击穿、缓存雪崩听起来都像“缓存失效了”，
> 但它们保护的根本不是同一件事。

如果这三个概念混在一起学，最后只会记成三句口号。
真正有价值的是搞清楚：

- 各自是什么流量形态
- 各自打穿的是哪一层
- 为什么这个项目对应采用了不同策略

### 1. 缓存穿透：查的是根本不存在的数据

穿透的关键词不是“热点”，而是：

**请求的目标在数据库里本来就不存在。**

例如：

- 用户传了一个根本不存在的店铺 ID
- 恶意请求不断构造不存在的 key

这时如果系统逻辑是：

- Redis 没有
- 去查 MySQL
- MySQL 也没有
- 下次还继续查

那么这些“无效请求”会稳定绕过缓存，直接打数据库。

这个项目处理穿透的方式很明确：

- 如果数据库查不到，就把空值写进 Redis
- 并且给这个空值一个短 TTL

也就是说，它承认：

**“不存在”本身也是一种可以缓存的结果。**

在代码里，空值缓存逻辑出现在 `queryWithPassThrough()` 中。

所以穿透主要保护的是：

- 数据库不要反复为“并不存在的数据”做无意义证明

### 2. 缓存击穿：查的是热点 key，而且刚好失效

击穿的关键词不是“不存在”，而是：

**一个真实存在、并且访问量很高的 key，在某个时刻失效了。**

例如：

- 店铺 1 是热点店铺
- 大量用户都在查它
- 结果这个 key 正好过期

这时问题就不是无效请求，而是大量真实请求在同一时刻一起回源。

所以击穿和穿透的根本区别是：

- 穿透：请求对象不存在
- 击穿：请求对象存在，而且很热

这个项目主要用两种思路处理击穿：

- 互斥锁回源
- 逻辑过期 + 后台重建

最终主方案选的是后者。

因为它真正想保护的是：

- 热点 key 失效时数据库不要被打爆
- 同时用户请求也尽量不要被大面积阻塞

所以击穿主要保护的是：

- 热点流量高峰下，数据库和接口延迟都要稳住

### 3. 缓存雪崩：不是一个 key 出问题，而是一批 key 一起出问题

雪崩和击穿最容易混。

击穿通常是：

- 一个热点 key 失效

雪崩通常是：

- 一大批 key 在同一时间集中失效
- 或者 Redis 整体不可用

比如：

- 很多缓存预热时用了完全相同的 TTL
- 到点后成片过期

或者：

- Redis 整体故障，缓存层整体失守

这时数据库面对的不是单个热点的回源流量，而是整个系统读流量的大面积回灌。

所以雪崩保护的不是某个 key，而是：

**整个缓存层不要在同一时间大面积失守。**

这个项目里虽然没有做完整的生产级雪崩治理，但能看到两个明显意识：

- 在部分方案里给 TTL 加随机值，避免同批 key 同时过期
- 对热点 key 更倾向于逻辑过期，而不是依赖真实 TTL 同时失效

这说明作者知道雪崩问题的本质是“过期时间同质化”。

### 4. 为什么这三个问题不能用同一种手段解决

因为它们打来的流量，性质根本不同。

穿透面对的是：

- 不存在的数据
- 重点是挡住无意义回源

击穿面对的是：

- 存在而且很热的数据
- 重点是热点 key 失效时保护数据库

雪崩面对的是：

- 多 key 或整个缓存层同时失守
- 重点是系统级流量回灌

所以策略自然不同：

- 穿透：空值缓存、布隆过滤器
- 击穿：互斥锁、逻辑过期、热点预热
- 雪崩：随机 TTL、多级缓存、限流降级、缓存高可用

如果把三者都简化成“缓存失效”，你就会误以为一个空值缓存能解决所有问题，这显然不对。

### 5. 在这个项目里，你应该怎么对应记忆

用“店铺详情”这个场景去记最清楚：

- 查一个根本不存在的店铺 ID  
  这是穿透

- 查一个大家都在访问的热门店铺，key 刚好过期  
  这是击穿

- 大量店铺缓存因为 TTL 设计不合理，在同一时刻一起过期  
  这是雪崩

一旦你能把这三个业务画面区分开，后面的技术策略就不容易混。

### 这一轮最该记住的一句话

这三个概念的区别，不在于“缓存有没有失效”，而在于：

**到底是不存在的数据绕过缓存、单个热点 key 失守，还是整层缓存同时失守。**
