这一轮只讲社交流的一个核心问题：

> 为什么这里不能只靠数据库把博客、关注、点赞查出来，
> 而要专门引入 Set、ZSet 和收件箱 Feed 这套设计。

如果不先把这个问题想清楚，看到代码时很容易误以为：

- 点赞只是加一减一
- 关注只是多一张关系表
- Feed 只是多一个查询接口

但这三个能力放在一起后，系统面对的已经不是简单 CRUD，而是：

**“一个人发内容以后，怎么让关注他的人高效看到；一个人又怎么快速判断自己和内容的关系。”**

### 1. 社交流的核心，不是“存内容”，而是“维护人与内容的关系”

在店铺点评业务里，博客内容本身当然可以放在数据库里。

真正难的不是存博客，而是这些实时关系：

- 我有没有点赞这篇博客
- 哪些人最近点过赞
- 我关注了谁
- 我和另一个用户的共同关注是谁
- 我关注的人刚发了什么，我怎么按时间往下刷

这些问题一旦多起来，数据库表虽然也能做，但代价会迅速上升：

- 高频查询
- 集合运算
- 时间排序
- 滚动分页

所以这个项目在社交流上真正做的事，不是“把博客放进 Redis”，而是：

**把人与内容、人与人之间的在线关系状态放进 Redis。**

### 2. 点赞为什么用 ZSet，而不是只改数据库字段

博客点赞在数据库里当然也有 `liked` 计数，但这还不够。

系统还要回答两个实时问题：

- 当前登录用户是否点过赞
- 最近点赞的前几个人是谁

如果只靠数据库计数字段，你只能知道总数，不知道是哪些人点的，更不知道顺序。

所以这里额外用 `blog:liked:{blogId}` 这个 ZSet：

- member 是用户 ID
- score 是点赞时间戳

这样就同时获得了三种能力：

- 判断是否点赞：查 score 是否存在
- 取消点赞：删除 member
- 查最近点赞用户：按分数顺序取前几名

也就是说，点赞在这里不是“给博客表 liked + 1”这么简单，
而是维护一份“内容和用户之间的实时关系索引”。

### 3. 关注为什么用 Set，而不是每次都去关系表算

关注关系在数据库里也会落表，但项目同时把它放进了 Redis Set。

原因不是为了少查一条 SQL，而是因为社交关系很快就会出现集合问题：

- 我是否关注了某人
- 我和你共同关注了谁

尤其是共同关注，这本质上就是集合交集。

Redis Set 对这种问题天然合适：

- follow A：往集合里加一个用户 ID
- unfollow A：从集合里删掉
- common follow：直接做交集运算

这里你应该看到的不是“Redis 很快”，而是：

**业务问题本身就是集合问题，所以它需要集合型数据结构。**

### 4. Feed 为什么不用普通分页，而要做“收件箱推送”

这部分最值得学。

很多人第一反应是：

- 我关注了很多人
- 直接查“我关注的人发的博客”
- 按时间排序分页不就行了

这个思路在关注量和内容量很小时可以工作。
但一旦数据量上来，它会变成一个典型的高成本查询：

- 先查关注列表
- 再查这些人的内容
- 再按时间排序
- 再做分页

而这个项目换了思路：

- 用户发博客时，不是等粉丝来查
- 而是主动把博客 ID 推到每个粉丝的收件箱里

这个收件箱就是 `feed:{userId}` 对应的 ZSet：

- member 是 blogId
- score 是发布时间戳

这样粉丝拉 Feed 时，不是做全局查询，而是直接从自己的收件箱里读。

这就是典型的“推模式”收件箱，而不是“拉模式”聚合查询。

它解决的核心问题是：

**把“读取时做大查询”改成“写入时做关系分发”。**

### 5. 为什么 Feed 要用滚动分页，而不是普通 pageNum

Feed 是按时间流动的，用户不断下拉，期间还可能有新内容插入。

如果这里用传统的页码分页：

- 第 1 页、第 2 页、第 3 页

一旦中间插入新数据，后面的页码就会漂移：

- 可能重复
- 可能漏数据

所以这里改成按 `lastId + offset` 的滚动分页。

本质上，它不是在说“第几页”，而是在说：

- 上一次我读到哪个时间点
- 这个时间点上已经消费了几条

这正是时间流式内容更适合的分页方式。

所以社交流这一块真正体现出的不是 Redis 技巧，而是：

**分页模型必须跟数据流动方式匹配。**

### 6. 这套社交流方案为什么适合当前阶段

它的优点是非常明确的：

- 点赞、关注、共同关注都能快速响应
- Feed 拉取不依赖复杂聚合查询
- 业务链路清楚，适合单体项目承载

它的代价也同样明确：

- 发一篇博客时要给所有粉丝做一次写扩散
- 用户粉丝量很大时，写入成本会上升
- 收件箱数据会膨胀，需要后续考虑裁剪和归档

所以这套方案不是“永远最优”，而是：

**在当前阶段，用写扩散换读性能和实现简单度。**

这很适合课程型项目，也适合中小规模社交流。

### 这一轮最该记住的一句话

这个项目的社交流设计，本质上不是“博客加点赞和关注”，而是：

**用 Redis 维护人与内容、人与人之间的实时关系，并把首页 Feed 从查询问题改写成收件箱读取问题。**
