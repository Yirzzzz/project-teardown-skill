### 上下文压缩

这一轮只讲一个核心问题：

**为什么这个项目不能靠“删掉旧消息”来解决上下文膨胀，而必须做分层的上下文压缩。**

这一轮只展开 4 个支撑点：

- 上下文为什么会失控
- 简单删历史为什么不够
- 这个项目实际采用了哪几层压缩机制
- 这些机制各自保护了什么

先暂时不展开的相关内容：

- Session Memory 的长期提炼与抽取策略
- Prompt cache break detection 的完整检测链
- REPL 内部消息存储与恢复细节

因为这些内容会影响压缩效果，但如果现在一起讲，会把主问题冲散。

#### 先把问题讲清楚：这个项目的上下文为什么特别容易爆

普通聊天应用的上下文压力主要来自人类文本，但这个项目不是。

这个项目的上下文压力主要来自三类内容：

- 工具结果，尤其是 `Read`、Shell、`Grep`、`Glob`、`WebFetch`、`WebSearch`
- 长会话消息链，REPL 会持续累积用户消息、助手消息、工具调用、权限消息、摘要边界
- 被重新注入的工作状态，比如文件附件、计划文件、技能内容、异步任务状态

也就是说，这里的上下文膨胀不是“聊天变长”这么简单，而是“工具化 Agent 会不断把外部世界搬进对话里”。

如果没有压缩，系统会遇到三个直接问题：

- Token 很快逼近模型窗口，后续请求无法继续
- 即使还能发请求，成本和延迟也会越来越高
- 旧工具结果会把真正有用的近期工作状态淹没，模型继续工作的效率会下降

#### 为什么不能简单删掉旧历史

直觉上看，最简单的方法是删最老的消息。但这个项目里这样做不够，原因有三层。

第一层，是**语义连续性**。

旧消息里不只是聊天，还包含用户真实意图、已经做过的修改、失败过的尝试、工具结果对应的结论。直接删除，很容易让模型不知道“现在为什么在做这件事”。

第二层，是**消息结构约束**。

这里的消息不是纯文本，而是带 `tool_use` / `tool_result` 配对关系的结构化消息。`sessionMemoryCompact.ts` 里专门有 `adjustIndexToPreserveAPIInvariants()` 来防止把成对消息切断。否则会出现保留了 `tool_result`，却把前面的 `tool_use` 删掉的情况，API 结构就会不合法。

第三层，也是最关键的一层，是**prompt cache 约束**。

从 `microCompact.ts` 的注释能看出来，这个系统非常在意缓存前缀是否还能复用。因为一旦你直接改写前缀，后续请求的 cache key 就变了，原本能命中的缓存会整体失效。  
所以这个项目面对的真实矛盾不是“删不删旧内容”，而是：

**既想让旧工具结果从未来请求里尽量消失，又不想因为删除动作本身破坏缓存前缀。**

这就是它必须做分层压缩的根本原因。

#### 这个项目实际用了三层压缩，而不是一把梭

从当前恢复树里看，至少能确认三层：

##### 1. Microcompact：先处理“旧工具结果太重”这个最高频问题

`microCompact.ts` 处理的是最常见、也最适合先动刀的膨胀源：旧工具结果。

这里又分成两种路径：

- `cached microcompact`
- `time-based microcompact`

`cached microcompact` 的本质不是摘要，也不是本地删消息，而是通过 `cache_edits` 一类机制，让旧工具结果在 API 层被逻辑删除，同时尽量保住前缀缓存。

这个设计解决的是：

- 工具结果重复占上下文
- 又不想因为改消息正文导致整段前缀缓存失效

所以它保护的是：**缓存命中率 + 上下文容量**。

`time-based microcompact` 则处理另一种情况：缓存已经冷了。  
代码注释写得很直白，如果距离上次 assistant 消息时间太久，服务器缓存大概率已经过期，那就没必要再执着于“无损缓存编辑”，可以直接把旧 `tool_result` 的内容替换成 `[Old tool result content cleared]` 这种占位文本，只保留最近少量结果。

这个设计保护的是：**在冷缓存场景下，优先缩小下一次真正要重写的 prompt 体积。**

你可以把 microcompact 理解成：

**不是压整段会话，而是优先把最胖、最容易过时、最影响上下文的旧工具结果瘦身。**

##### 2. Compact：当整体会话还是太大时，再做“摘要化重构”

如果轻量清理还不够，就进入 `compact.ts` 这一层。

这一层做的事就不是“局部清空工具结果”了，而是：

- 选出要被总结的历史消息
- 给模型一个专门的 compact prompt
- 让模型生成结构化摘要
- 用 `compact boundary + summary + 少量保留消息 + 附件` 的形式重建后续上下文

这里最值得注意的不是“它会摘要”，而是**它不会把所有东西都交给摘要**。

它在压缩后还会补一些关键附件回来，比如：

- 最近读过的文件内容
- 当前 plan 文件
- 已调用过的技能内容
- 异步 agent 的任务状态

这说明它很清楚一件事：

**摘要可以保留叙事，但不能完全替代工作现场。**

如果只留下摘要，模型往往知道“做过什么”，却不知道“当前文件状态是什么、接下来该接哪条线”。  
所以 compact 的本质不是普通总结，而是：

**把旧会话改写成一个可继续工作的最小工作包。**

##### 3. Session Memory Compact：尽量不用重新总结整段历史

`sessionMemoryCompact.ts` 代表第三层思路：如果系统已经有更长期、更结构化的 session memory，那就不一定每次都重新总结全部旧历史。

它做法更像：

- 用已有的 session memory 作为“长期摘要”
- 只保留最近一段必须保真的消息
- 再补上 plan 等关键附件

为什么这很重要？

因为如果每次都重新跑一次大摘要，成本高、波动大，而且摘要质量会被最近噪音拖累。  
有了 session memory 以后，系统可以把“长期事实”和“近期工作现场”拆开管理。

它保护的是：**压缩成本、稳定性，以及长会话的持续可工作性。**

#### 这三层机制分别在保护什么

如果从架构视角总结，可以把它们看成三道闸：

- `microCompact`：保护近期请求不要被旧工具结果拖垮
- `compact`：保护整段会话在接近窗口上限时还能继续
- `session memory compact`：保护超长会话不要每次都重新付出整段摘要成本

这三层不是重复设计，而是在处理三种不同张力：

- 局部内容太胖
- 整体会话太长
- 长期运行成本太高

#### 一个更本质的理解：它压的不是“历史”，而是“未来请求的负担”

这是我觉得最值得学走的地方。

很多人理解上下文压缩，会停留在“把旧内容缩短”。  
但从这套实现看，这个项目真正关心的是：

- 下一个请求还能不能发
- 下一个请求要花多少 token
- 下一个请求还能不能命中 prompt cache
- 下一个请求发出去后，模型还能不能继续当前工作

所以它压缩的目标并不是“让历史更短更好看”，而是：

**让未来每一次请求继续可用、可负担、可衔接。**

这比“做摘要”高一层。

#### 这一层为什么值得优先理解

因为它直接体现了 Agent 系统和普通聊天系统的区别。

普通聊天系统的压缩，大多是在做对话摘要。  
这个项目的压缩，是在同时处理：

- 工具结果膨胀
- API 结构约束
- prompt cache 稳定性
- 工作现场恢复
- 长会话持续运行

如果你把这一层吃透，就会明白为什么一个真正能干活的本地 Agent，迟早都会长出“多层上下文管理”，而不是只靠一个 summary 按钮。
