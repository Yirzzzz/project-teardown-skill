### Compact 主流程深拆

这一轮只讲一个核心问题：

**`compact` 表面上是在“生成摘要”，但它为什么实际上更像一次会话重构，而不是一次总结。**

这一轮只展开 4 个支撑点：

- 它先如何选“能被压”的那部分消息
- 它如何向模型发起一次专门的 compact 请求
- 它如何把摘要翻译成新的会话结构
- 它为什么还要维护边界和保留段的连接关系

先暂时不展开的内容：

- Session memory compact 这条替代路径
- reactive compact 的触发条件
- prompt cache sharing 的性能细节

因为这一轮最重要的是先把 `compact` 自己的结构讲清楚。

#### 第一步：它不是随便截断，而是先找“API 安全的可切分段”

从 `grouping.ts` 能看出来，这个项目在切会话时非常克制。

它不是简单按“最近 N 条消息”切，也不是按“最近几轮人类发言”切，而是按 **API round** 分组。  
也就是：以一次新的 assistant 响应开始作为边界，把与之相关的 `tool_use`、`tool_result`、后续 assistant chunk 视为同一组。

这样做的原因非常现实：

- 这里的消息不是平铺文本
- 里面有工具调用和工具结果的结构约束
- 如果切分切到了半个 round 中间，后面很容易出现 `tool_result` 找不到对应 `tool_use`

所以 `compact` 的第一步其实是在回答：

**什么地方切下去，后面的消息结构还是合法的。**

这就是为什么它先做 grouping，而不是先做 summarization。

#### 第二步：它不是拿原对话让模型“自由总结”，而是发起一条专用 compact 请求

`compact.ts` 里会先构造 `compactPrompt`，再包成一条 `summaryRequest` 发给模型。  
`prompt.ts` 里的 prompt 也说明得很清楚：模型不是普通聊天，而是被要求做一件非常特定的事：

- 只能输出文本
- 先写 `<analysis>` 草稿
- 再写 `<summary>`
- 要覆盖用户请求、关键技术点、文件、错误、当前工作、下一步等结构化栏目

这说明 `compact` 不是随手让模型“帮我总结一下”，而是在调用一个专门的摘要子流程。

更重要的是，它在请求前还会做几件事：

- 执行 pre-compact hooks
- 合并用户提供的额外 compact 指令
- 在必要时对 prompt-too-long 做重试与截断

也就是说，它其实先把“要怎么总结”这件事变成一条受控的、可恢复的内部任务，而不是临场发挥。

#### 第三步：摘要生成后，它不会把原会话替换成“一段摘要”，而是重建成一条新链

这是 `compact` 最容易被误解的地方。

很多人会想象它做的是：

- 丢掉旧历史
- 留下一段 summary

但从 `buildPostCompactMessages()` 和返回的 `CompactionResult` 看，真正的结果是：

- `boundaryMarker`
- `summaryMessages`
- `messagesToKeep`
- `attachments`
- `hookResults`

这其实是一条新的、重组过的消息链。

每一部分都在承担不同角色：

- `boundaryMarker`：声明“这里发生过一次压缩”，同时记录压缩元信息
- `summaryMessages`：把旧历史转译成模型还能理解的紧凑叙事
- `messagesToKeep`：保留近期还不能丢的原文现场
- `attachments`：回填继续工作必需的状态
- `hookResults`：把 session start / post compact 这类系统级上下文重新接回来

所以 `compact` 的本质不是“把历史变短”，而是：

**把旧会话重新编译成一个更短、但仍然能跑的上下文包。**

#### 第四步：为什么还要有 boundary，而不是直接“summary 接着 recent messages”

因为一旦压缩发生，系统不只是内容变了，消息链本身也变了。

`boundaryMarker` 的作用至少有三层：

##### 1. 明确告诉后续系统：这里之前已经压过

后面的逻辑经常会用 `isCompactBoundaryMessage()` 或 `getMessagesAfterCompactBoundary()` 来判断“现在看到的是完整原始会话，还是已经压缩过的会话尾部”。  
如果没有边界，后续再压、再恢复、再 resume 时就很难知道从哪里开始算“当前活跃上下文”。

##### 2. 给摘要和保留段之间提供一个稳定锚点

`annotateBoundaryWithPreservedSegment()` 会把保留段的：

- `headUuid`
- `anchorUuid`
- `tailUuid`

记到 boundary 里。

这说明它不是只想把内容拼起来，而是还想告诉后续加载逻辑：

**新链和旧链之间，哪些消息被保留了，它们应该从哪里重新接上。**

换句话说，boundary 既是“压缩发生过”的标记，也是“消息链重连说明书”。

##### 3. 防止后续压缩或恢复时把旧边界搞乱

你在 partial compact 相关逻辑里也能看到，某些方向的压缩必须显式过滤旧的 compact summary / boundary，否则 backward scan 会被旧边界误导，导致新 summary 失效或保留段被错接。

这说明 boundary 不是装饰，而是后续一切“继续压、继续恢复、继续加载”的基础设施。

#### `messagesToKeep` 为什么不是可有可无

在 `partial compact` 路径里，这一点尤其明显。

代码先决定：

- 哪部分要总结 `messagesToSummarize`
- 哪部分保留原文 `messagesToKeep`

这说明作者很明确地承认：

**摘要无法完全替代最近一段原始消息。**

原因通常有三个：

- 最近消息里包含精确语气和最新用户反馈
- 最近消息里可能有还没沉淀进摘要的工具现场
- 最近消息往往是模型“当前正在做的动作”的直接前文

所以 `compact` 从来不是“全部历史摘要化”，而是更像：

**把旧的部分摘要化，把新的部分保真化。**

这也是为什么在 partial compact 里，还会根据方向决定：

- `up_to`：总结前缀，保留后缀
- `from`：总结后缀，保留前缀

它不是只有一种压法，而是在做“围绕某个 pivot 的会话重排”。

#### 这整套流程最本质的地方：它像编译，而不是像摘要

如果把这条链路压成一句话，我会这样说：

`compact` 不是对旧对话做一份笔记，而是在把旧对话重新编译成一个新的可执行上下文。  

“可执行”这三个字很重要，因为它强调的不是信息保存，而是后续工作能否继续。

所以它的流程才会是：

1. 先找安全切点
2. 再发专门摘要请求
3. 再重建新的上下文包
4. 再用 boundary 和 preserved segment 维护新旧链关系

如果少任何一步，它都更像普通总结工具；  
正因为四步都在，它才更像一个会话运行时里的“上下文重构器”。

#### 当前确认与推断的边界

已确认的内容：

- `compact` 先按 API round 组织消息，而不是直接按条数截断
- 它使用专门的 compact prompt 来生成结构化 summary
- 压缩结果是 `boundary + summary + keep + attachments + hooks` 的新结构
- boundary 中还会标注 preserved segment 的 relink 信息

基于实现的合理推断：

- 这套设计的优先级不是“摘要写得像不像人”，而是“压缩后的会话结构是否还能被后续系统正确消费”

如果下一轮继续拆，最值得看的就是：

**`preserved segment relink` 到底在防什么结构性错误。**

因为那一层会进一步回答：

**为什么消息链一旦压缩过，后续恢复就不能只靠内容相邻。**
