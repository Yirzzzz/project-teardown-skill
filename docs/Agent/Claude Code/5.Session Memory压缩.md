### Session Memory Compact vs 传统 Compact

这一轮只讲一个核心问题：

**`session memory compact` 和传统 `compact` 看起来都在压缩上下文，但它们到底是在解决同一个问题，还是两种不同思路。**

先给结论：

它们不是前后步骤关系，而是两条不同的重型压缩路径。  
`session memory compact` 更像“优先利用已经沉淀好的长期记忆”，  
传统 `compact` 更像“现场重新总结旧会话，再重建上下文包”。

#### 先看它们在调用链里的位置

在 `autoCompactIfNeeded()` 里，顺序非常明确：

1. 先判断是否需要重型压缩
2. 先尝试 `trySessionMemoryCompaction()`
3. 如果成功，直接返回
4. 如果失败或条件不满足，才调用 `compactConversation()`

所以关系不是：

- `compact` 做完之后，最后再做 `session memory compact`

而是：

- **先试 session memory compact，试不通再退回传统 compact**

这意味着系统作者把 `session memory compact` 当成一种“优先路线”，不是一个尾处理步骤。

#### 两者的核心差别：摘要来源不同

这是最根本的一点。

传统 `compact` 的摘要来源是：

- 把当前旧会话送给模型
- 现场生成一份新的结构化摘要

而 `session memory compact` 的摘要来源是：

- 先读取已经存在的 session memory
- 如果 session memory 不是空模板，就直接拿它作为长期摘要基础

所以它们最大的差别其实不是“压多少”，而是：

- 传统 `compact`：**当场重新总结**
- `session memory compact`：**优先复用已有长期记忆**

这会带来后面一连串不同。

#### 传统 Compact 更像一次“在线重构”

传统 `compact` 的思路是：

- 现场挑出要总结的消息
- 发一个专门的 compact prompt 给模型
- 让模型生成结构化 summary
- 再重建出 `boundary + summary + keep + attachments`

所以它特别适合：

- 还没有成熟 session memory
- 当前会话的近期细节很多，必须临时重总结
- 需要一个高度贴合当前上下文状态的摘要

但它的代价也明显：

- 要额外发一次 compact 请求
- 成本更高
- 输出质量会受当前会话噪音影响
- 如果 compact 请求自己又太长，还得走 PTL retry

所以传统 `compact` 更像：

**为了把当前旧会话重新组织起来，现场再跑一次摘要编译。**

#### Session Memory Compact 更像一次“长期记忆 + 近期窗口”的拼接

`sessionMemoryCompact.ts` 的做法明显不同。

它会先看：

- session memory 功能是否启用
- session memory 文件是否存在
- 内容是不是空模板

如果这些条件成立，它就尽量不再重跑整段摘要，而是：

- 用 session memory 作为长期摘要
- 再保留最近一段必须保真的消息
- 再补 plan 等关键附件

也就是说，它的整体思路是：

**长期事实靠 session memory，近期现场靠 messagesToKeep。**

这比传统 `compact` 更“分层”。

#### 为什么系统会更想优先试 session memory compact

因为从工程角度看，它通常更便宜，也更稳定。

原因有三个：

##### 1. 不必每次重新总结整段旧会话

如果一段长会话每次逼近阈值都要重新总结一次，成本高，而且摘要质量受瞬时状态影响很大。  
session memory 让系统可以把长期部分提前沉淀掉，避免反复为同一段历史付费。

##### 2. 长期事实和近期工作可以分开管理

传统 `compact` 是把大量历史重新揉成一份 summary。  
而 `session memory compact` 天然更适合这种结构：

- 长期约束、偏好、背景信息放 session memory
- 最近正在做的具体工作保留在 keep 段

这更像一个成熟 Agent 系统会采用的分层上下文模型。

##### 3. 更容易控制 post-compact 的体积

`sessionMemoryCompact.ts` 里会根据配置决定：

- 至少保留多少 token
- 至少保留多少含文本消息
- 最多保留多少 token

也就是说，它在“长期记忆已存在”的前提下，更容易把保留窗口控制在一个可预测范围内。

#### 那为什么还需要传统 compact，不能只靠 session memory

因为 session memory 不是总能用，也不是总够用。

代码里已经体现了很多失败分支：

- session memory 功能没开
- session memory 文件不存在
- 内容只是空模板
- 找不到 last summarized message 的边界
- 压完后体积仍然超过 autoCompactThreshold

这说明系统作者很清楚：

**session memory 是很好的长期记忆层，但它不是任何时候都能替代当前会话的现场重总结。**

换句话说：

- session memory compact 是“有沉淀时优先利用沉淀”
- 传统 compact 是“沉淀不够时的兜底重构器”

#### 两者真正的分工

如果把它们放到一张架构图里，可以这么理解：

- `microcompact`：先减掉最胖的旧工具结果
- `session memory compact`：优先用长期记忆替代重复总结
- 传统 `compact`：当长期记忆不可用或不够时，重新把旧会话编译成新上下文

这三者并不是谁淘汰谁，而是分层分工：

- 轻量清理
- 长期记忆复用
- 现场摘要重构

#### 一个更重要的架构含义

这组设计说明这个项目已经不把“压缩”理解成单一动作了。

它在区分至少三种不同来源的上下文价值：

- 短期但过时的工具结果
- 长期稳定的会话记忆
- 当前仍在施工的近期现场

不同来源，就对应不同压缩方式。

这就是为什么你会同时看到：

- `microCompact`
- `sessionMemoryCompact`
- `compactConversation`

它们不是重复造轮子，而是在回应不同层级的上下文问题。

#### 当前确认与推断的边界

已确认的内容：

- `autoCompactIfNeeded()` 中会先尝试 `trySessionMemoryCompaction()`，失败后才走传统 `compactConversation()`
- `session memory compact` 直接复用已有 session memory，而不是重新发起 compact 摘要请求
- 它同样会保留一段 `messagesToKeep` 并补 plan 附件

基于实现的合理推断：

- 系统作者希望把“长期历史压缩”尽量前置为 session memory，而把传统 compact 留给更动态、更临场的重构场景

如果下一轮继续拆，最值得看的就是：

**`preserved segment relink` 为什么必须存在。**

因为那一层最能解释：

**压缩后为什么消息链恢复不是“数组拼接”这么简单。**
