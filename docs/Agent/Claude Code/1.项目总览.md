# Claude Code 项目拆解

> Skill usage marker: used the project-teardown skill

## Part One: 当前全局概览

### 一句话判断

这是一个以终端交互为外壳、以“大模型主循环 + 本地工具执行 + 动态能力装配”为核心的 Agent CLI 平台，而不只是一个普通的命令行工具。

### 项目本质

从代码结构看，这个仓库的本体是一个恢复出来的 `Claude Code` 源码树。它的目标不是单纯解析命令参数，而是把下面几类能力拼成一个统一运行时：

- 终端 UI / REPL：用户在 TUI 中输入、查看消息、审批权限、切换模式
- Agent 主循环：把用户输入、系统上下文、工具集合、MCP 能力组装成一次模型请求
- 本地执行层：文件读写、Shell、计划模式、任务、子代理等工具真正落地执行
- 扩展能力层：技能、插件、工作流、MCP server、远程控制、后台任务等按需接入

所以它本质上更像“可扩展的本地 Agent 操作系统前端”，而不是传统的单进程 CLI。

### 它解决的核心业务痛点

这个项目试图解决的是：如何让一个在本地代码仓里工作的 AI 助手，既能理解上下文，又能安全地调用工具，还能在能力不断增长时不把系统做成一团不可维护的硬编码。

核心痛点主要有四个：

- 用户需要一个能直接在仓库里工作的 AI，而不是只能聊天的网页助手
- 模型必须调用本地工具，但工具权限、模式切换、远程接入都带来复杂度
- 能力很多，不能把所有功能静态塞进主入口，否则启动和维护成本都会失控
- 这个产品明显存在内外部版本差异，因此需要 feature gate、用户类型、远程配置一起控制功能暴露

### 一条核心请求流

如果只抓主干，一次典型请求大致这样流动：

1. 用户通过 `src/entrypoints/cli.tsx` 进入 CLI，先走一层 fast-path 分流
2. 普通交互路径进入 `src/main.tsx`，在这里完成全局初始化、配置读取、命令装配、工具装配、远端能力预热
3. `src/replLauncher.tsx` 把 `App` 和 `REPL` 挂起来，真正进入终端交互界面
4. `src/screens/REPL.tsx` 维护会话主循环：收集输入、维护消息流、处理权限、调用查询逻辑、更新 UI 状态
5. 请求发出前，会把 `src/context.ts` 中的系统上下文和用户上下文拼进去，比如 git 状态、`CLAUDE.md`、日期等
6. 模型可见的能力不是固定的，而是由 `src/tools.ts` 和 `src/commands.ts` 动态汇总出来，包括内建工具、MCP 工具、技能、插件命令、工作流命令
7. 执行结果再回到 `REPL` 状态树中，驱动消息展示、任务状态、远程桥接状态和后续交互

如果你把这条链路看懂，这个项目的大框架就已经抓住了。

### 这个项目最核心的架构思想

我认为它最核心的架构思想不是“命令系统”，而是：

**把 AI 助手做成一个可组合运行时，而不是做成一堆散落功能。**

具体表现为三点：

- 入口层强分流：先用 fast-path 处理高频或特殊模式，避免每次都加载整个大系统
- 能力层统一抽象：命令、工具、技能、MCP、插件最终都被折叠进统一的运行时里
- 状态层集中承接：`AppState` 把本地 UI、任务、权限、桥接、插件、MCP、计划模式等全压到一个共享状态模型上

这套设计解决的是“能力不断增加后系统如何不崩掉”的问题。

### 最值得先理解的 4 个核心机制

#### 1. 启动分流与主装配

为什么存在：这个项目非常重，若每次启动都完整加载 `main.tsx`，性能和复杂度都会很差。

它本质在做什么：在真正进入主程序前，先把 `--version`、bridge、daemon、background session、worktree/tmux 等路径单独短路。

如果没有这层：所有模式都会耦合到一个超级入口里，启动性能、维护性、隔离性都会明显变差。

#### 2. 命令 / 技能 / 插件 / 工作流统一命令面

为什么存在：用户看到的是一个 slash command 世界，但这些命令其实来源完全不同。

它本质在做什么：`src/commands.ts` 把内建命令、技能目录、插件命令、工作流命令、动态技能统一收口，再根据可用性和 gate 做过滤。

如果没有这层：每增加一种扩展来源，都要重写一套发现、展示、调用逻辑，系统会快速失控。

#### 3. 工具池组装与权限过滤

为什么存在：模型能调用哪些工具，不能只靠“是否注册”，还要结合权限模式、REPL 模式、MCP、用户类型一起决定。

它本质在做什么：`src/tools.ts` 先定义基础工具全集，再根据模式和 deny rule 过滤，最后把内建工具和 MCP 工具组装成真实可用工具池。

如果没有这层：模型视角和运行时视角会不一致，权限与执行边界很容易出错。

#### 4. REPL 作为统一会话内核

为什么存在：这个项目不是一次性命令执行，而是持续会话，里面还混合了消息流、任务、审批、远程桥接、插件状态、计划模式等复杂交互。

它本质在做什么：`src/screens/REPL.tsx` 充当统一交互内核，把 UI、消息、模型调用、工具执行、恢复、后台任务连接起来。

如果没有这层：每个模式都得自己维护状态和消息生命周期，功能越多越难协同。

### 当前大图总结

从大图上看，这个项目可以理解成四层：

- 入口与启动分流层：`src/entrypoints/cli.tsx`、`src/main.tsx`
- 交互与状态层：`src/replLauncher.tsx`、`src/screens/REPL.tsx`、`src/state/`
- 能力装配层：`src/commands.ts`、`src/tools.ts`、`src/skills/`、`src/plugins/`、`src/services/mcp/`
- 扩展场景层：`assistant`、`bridge`、`coordinator`、`proactive`、`voice`、`buddy` 等 feature-gated 模块

确认项：

- 这是一个 Bun + TypeScript + React Ink 的终端 Agent 应用
- 主架构围绕 REPL、工具池、命令池、AppState 展开
- 大量能力通过 feature gate 和用户类型按条件注入

基于代码的合理推断：

- 这个仓库明显同时服务“外部发布版”和“内部狗粮版”，因此架构上非常强调可裁剪性
- 长期演进压力不在单个命令，而在“能力增长后如何继续统一装配和控制”

当前未完全确认但值得以后验证的点：

- 真正的模型请求拼装细节在 `query`、prompt 构建和 compact 相关模块里，首轮还没有展开
- MCP 连接生命周期、插件生命周期、远程 bridge 的边界处理还没单独拆

### 建议下一轮优先深挖方向

下一轮最值得优先深挖的是：

**工具池与权限控制为什么必须这样设计。**

原因很简单：这个项目最难也最有工程价值的地方，不是 UI，也不是命令数量，而是“模型看到什么工具、何时能调、谁来审批、远程模式下怎么收敛边界”。这直接决定它为什么能从一个聊天 CLI 进化成真正可用的本地 Agent。

如果你更偏系统视角，后续也可以按这个顺序继续拆：

1. 工具池与权限控制
2. 命令 / 技能 / 插件 / MCP 的统一装配
3. REPL 主循环与消息状态管理
4. Bridge / 后台任务 / 多代理这些扩展场景如何挂到主系统
