### 上下文压缩的数据流

这一轮只回答一个问题：

**在这个项目里，一次上下文压缩从输入到输出，数据到底怎么流。**

如果把所有细节先压平，这条数据流可以看成 8 个阶段。

#### 1. 原始会话先进入主消息流

起点永远是当前会话消息数组，也就是 REPL 持有的那条对话链。  
里面不只是用户和助手文本，还包括：

- `tool_use`
- `tool_result`
- thinking
- attachment
- compact boundary
- compact summary

与此同时，系统还维护一些和压缩强相关的旁路状态：

- `readFileState`：最近读过哪些文件
- invoked skills：本轮或本 agent 调用过哪些技能
- `AppState.tasks`：后台 agent / 任务状态
- plan / plan mode 状态
- session memory

所以压缩的输入不是单一“消息数组”，而是：

**消息主链 + 若干工作状态缓存。**

#### 2. 先走轻量判断：能不能只做 microcompact

当一轮请求要发起时，系统会先看能不能只做局部减负。

这里输入的是：

- 当前 messages
- querySource
- 最近 assistant 的时间
- 当前模型 / 权限 / 查询来源

然后分流成两类：

- `cached microcompact`
- `time-based microcompact`

输出不是 summary，而是两种不同结果：

- 要么消息正文保持不变，但生成 `pendingCacheEdits`
- 要么直接把旧 `tool_result` 内容替换成清空占位文本

所以这一层的数据流本质是：

**旧工具结果分析 → 轻量删减指令或正文清空 → 回到主请求流。**

如果这一步已经足够减轻上下文负担，系统就不必进入重型压缩。

#### 3. 再走重型判断：是否需要 autocompact

如果轻量处理不够，系统才会开始看更重的压缩路径。

这里主要输入的是：

- 全量 messages 的 token 估算
- 当前模型的有效上下文窗口
- auto compact threshold
- 是否处于特定 querySource / 特殊模式

这一层的输出是一个布尔判断：

- 不需要重型压缩：继续正常请求
- 需要重型压缩：进入下一层分支

所以这里的数据流是：

**messages → token estimation → threshold decision → 选择是否触发重型压缩。**

#### 4. 重型压缩先尝试 session memory compact

一旦需要重型压缩，系统不会立刻进传统 `compact`。  
它会先试 `session memory compact`。

这里的数据输入包括：

- 当前 messages
- `lastSummarizedMessageId`
- session memory 文件内容
- 保留窗口配置

处理过程大致是：

- 先判断 session memory 是否存在且不为空模板
- 根据 `lastSummarizedMessageId` 找出哪些历史已被长期记忆覆盖
- 算出 `messagesToKeep`
- 构造新的 `boundary + summary + keep + plan attachment`

输出是一个 `CompactionResult`。  
如果成功，这条路径直接结束，不再进入传统 `compact`。

所以它的数据流是：

**已有长期记忆 + 当前消息尾部 → 直接拼出压缩后的新上下文。**

#### 5. 如果 session memory 不够，再进入传统 compact

这才是你前面一直在问的那条经典路径。

这里的数据输入包括：

- 待总结的消息段
- compact prompt
- hooks 注入的额外指令
- 允许共享 cache prefix 的参数

处理过程是：

1. 先确定 `messagesToSummarize`
2. 生成 `summaryRequest`
3. 通过 `streamCompactSummary()` 发起专门的 compact 请求
4. 得到原始 summary 文本

如果 compact 请求自己还太长，就会走 PTL retry，把更旧的消息组继续裁掉再重试。

所以这一层的数据流是：

**旧消息段 → compact prompt → 模型输出结构化 summary。**

#### 6. 摘要出来后，系统不会直接覆盖原消息，而是组装 `CompactionResult`

无论是传统 compact 还是 session memory compact，最终都会进入类似的数据结构：

- `boundaryMarker`
- `summaryMessages`
- `messagesToKeep`
- `attachments`
- `hookResults`
- 若干 token / usage 元数据

这里最关键的转换是：

- 原始旧会话，不再以“逐条消息历史”的形式存在
- 它被转译成一份新的、可继续工作的压缩上下文包

所以这一步的数据流是：

**summary / memory + keep + state attachments → `CompactionResult`。**

#### 7. `CompactionResult` 再被展开成新的消息链，回写到会话状态里

`buildPostCompactMessages()` 会把 `CompactionResult` 展开成实际的新消息序列：

- boundary
- summary
- keep
- attachments
- hook messages

随后 REPL / 会话状态会用这条新链替换旧链。

同时还会做一些压缩后的状态清理，例如：

- 清理旧的 `readFileState`
- 重置 microcompact 状态
- 清理部分缓存和 tracking
- 保留仍需跨压缩存活的状态，比如 invoked skills

所以这一步的数据流是：

**结构化压缩结果 → 新消息链 → 写回 AppState / session state。**

#### 8. 下一轮正常请求时，系统读取的是“压缩后的新上下文”

压缩完成后，下一轮模型请求看到的就不再是旧历史，而是：

- 新 boundary
- compact summary
- 最近保留消息
- 回填附件
- 重新注入的 session-start / system-level 上下文

如果是 cached microcompact，还会在 API 组包阶段继续把：

- `cache_edits`
- `cache_reference`

插进消息参数里。

所以最后一跳的数据流是：

**压缩后的消息链 + 压缩后的 API 组包规则 → 下一轮模型请求。**

#### 如果把它压成一张最小流程图

可以这样记：

1. 原始消息链进入请求前处理
2. 先尝试 microcompact 减掉旧工具结果
3. 如果还不够，再做重型压缩判断
4. 优先尝试 session memory compact
5. 不行再走传统 compact summary
6. 得到 `CompactionResult`
7. 展开成新的消息链并回写会话
8. 后续所有请求都基于这条新链继续

#### 这条数据流最值得你记住的地方

不是“系统有很多压缩模块”，而是：

**压缩并不是请求链路外的一次旁路优化，它本身就是请求链路的一部分。**

换句话说，这个项目里的上下文压缩不是：

- 先聊天
- 再偶尔总结

而是：

- 每次请求前先判断上下文压力
- 必要时改写上下文本身
- 再把改写后的上下文送给下一轮模型

这就是为什么它的数据流看起来更像“编译管线”，而不是一个单独功能点。
