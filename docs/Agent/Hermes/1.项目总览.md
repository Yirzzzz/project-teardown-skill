# Hermes Agent 架构拆解

> Skill usage marker: used the project-teardown skill

## Part One: Current Global Overview

### 一句话判断

这是一个把“大模型对话能力”包装成“长期运行的通用 Agent 平台”的项目，不只是 CLI 聊天工具，而是一个统一承载多入口、多工具、多会话记忆、多运行环境和多平台代理的 Agent 操作系统雏形。

### 项目本质

从代码结构看，Hermes Agent 的核心不是某个单独模型，也不是某个单独工具，而是一个同步的 Agent 编排内核：

- 上游接多种入口：CLI、消息平台 gateway、ACP 编辑器接入、batch runner、RL 环境。
- 中间用 `AIAgent` 维护一次完整的对话循环、上下文、工具调用和压缩逻辑。
- 下游接统一工具体系：terminal、file、web、browser、memory、delegate、MCP、cron、messaging 等。
- 外围再补上持久化、技能系统、配置系统、主题皮肤、环境隔离和多 profile 支持。

它更像“一个可嵌入到不同交互壳里的 Agent runtime”，而不是“一个单点功能应用”。

### 它解决的核心业务痛点

这个项目试图解决的不是“让模型回答问题”，而是“让模型能长期、稳定、跨场景地替用户做事”。这会自然带来几类工程痛点：

1. 模型本身没有执行能力，所以需要统一工具层把搜索、读写文件、跑命令、浏览器操作、调度任务这些动作接进来。
2. 单轮对话不够，复杂任务会跨多轮、多会话、多平台，所以需要会话存储、记忆、技能和历史搜索。
3. 上下文会膨胀，工具输出尤其容易把 token 撑爆，所以必须有上下文压缩和 prompt cache 友好的策略。
4. 一个 Agent 不只跑在本地终端，还可能跑在 Telegram、Discord、Slack，甚至远程容器和 serverless 环境里，所以需要统一的 gateway 和环境抽象。
5. 用户既想要能力强，又想要成本和安全边界可控，所以项目引入 toolset、approval、profile、多环境后端和配置分层。

### 一条核心请求是怎么流动的

可以把最典型的一次请求理解成下面这条主链路：

1. 用户从 CLI 或消息平台发来请求。
2. 入口层把请求变成 Hermes 内部的 session/context 形式。
3. `run_agent.py` 里的 `AIAgent` 组装 system prompt、技能提示、记忆、上下文文件、会话历史和工具定义。
4. Agent 调用模型。
5. 如果模型返回 tool call，就经由 `model_tools.py` 和 `tools/registry.py` 分发到具体工具。
6. 工具结果再回填到消息序列里，Agent 继续下一轮，直到模型给出最终回答。
7. 会话过程被写入 `hermes_state.py` 的 SQLite 状态库；如果上下文过长，则由 `agent/context_compressor.py` 压缩。
8. 如果入口是 gateway，最终结果再被路由回原平台；如果是 CLI，则由 `cli.py` 负责交互显示。

这条链路说明它的架构重点不是“单次推理”，而是“把模型调用、工具调用、状态管理、交互入口、长期记忆串成一个可反复运行的闭环”。

### 最核心的架构思想

我认为这个项目最核心的架构思想是：

**把 Agent 设计成一个稳定的中枢循环，然后把所有不稳定、易变化的能力都外插成模块化边界。**

具体体现为：

- 模型能力可切换：不同 provider、不同 base URL、不同上下文长度。
- 工具能力可插拔：每个工具自注册到 registry，再由 toolset 控制暴露范围。
- 交互入口可替换：CLI、gateway、editor integration 共用同一个 Agent 内核。
- 运行环境可替换：local、docker、ssh、modal、daytona、singularity。
- 长期能力可累积：memory、session search、skills、trajectory、cron。

如果没有这种“中枢循环 + 周边模块”的设计，这个项目很快就会退化成大量平台逻辑、工具逻辑和模型适配代码彼此缠绕的巨石脚本。

### 当前最值得先理解的 4 个机制

#### 1. Agent 主循环为什么是整个系统的“心脏”

`AIAgent` 把模型调用、tool call 回填、iteration budget、压缩、提示词组装、失败回退这些能力放在一个中心循环里。  
这解决的是“无论从哪个入口进来，最终都能走一套一致的 Agent 执行语义”的问题。

如果没有这一层，CLI、gateway、batch、RL 环境都会各自复制一套 agent 行为，后面会非常难维护。

#### 2. 工具注册表 + toolset 为什么是平台扩展的关键

`tools/*.py` 在导入时注册 schema、handler、可用性检查；`model_tools.py` 负责发现工具并按 toolset 暴露给模型。  
这解决的是“工具越来越多时，如何保持统一分发、统一可见性控制、统一 availability 检查”的问题。

如果没有 registry，工具系统会变成大量 if/else 分发，MCP 和插件型扩展也很难接入。

#### 3. 上下文压缩为什么是 Agent 项目的硬约束

Hermes 不只是聊天，还会积累工具结果、历史消息、技能提示、记忆块、平台上下文。  
`ContextCompressor` 的存在说明团队已经把“上下文膨胀”当成一等公民问题，而不是后期优化项。

这层机制解决的是：复杂任务进行到中后段时，如何在不直接把系统弄崩、也不让成本无限上涨的前提下继续工作。

#### 4. SQLite 状态库 + session search 为什么是“长期性”的基础

`hermes_state.py` 用 SQLite + FTS5 管会话和消息，不是为了炫技，而是因为这个 Agent 明显在往“跨会话连续体”方向做。  
如果历史只是 JSONL 文件，很快就无法稳定支持搜索、压缩分裂后的 session 关联、成本统计和跨平台查询。

这说明项目已经不满足于“当前窗口里像个助手”，而是试图成为“长期记住上下文并可被再次调用的代理体”。

### 当前大图景总结

从整体上看，这个项目可以拆成五层：

1. 交互层：CLI、gateway、ACP、batch、RL。
2. Agent 编排层：`AIAgent`、prompt builder、context compressor、usage/pricing、display。
3. 能力层：tool registry、toolsets、MCP、plugins、delegate、execute_code。
4. 状态层：SQLite session store、memory、skills、session search、trajectory。
5. 运行与隔离层：profiles、terminal backends、approval、安全策略、配置系统。

它现在采取的是“单体仓库中的模块化平台架构”，不是微服务。  
这是一个非常现实的阶段性选择：对于 Agent 产品，最先爆炸的通常不是服务拆分，而是 prompt、工具、上下文、状态和平台适配之间的耦合复杂度。Hermes 目前优先解决的是这类复杂度。

### 我对当前项目阶段的判断

基于仓库结构和入口代码，可以做两个判断：

- 可以确认的：这是一个功能已经相当广的成熟 Agent 平台代码库，重点在统一运行时，而不是单点模型实验。
- 合理推断的：项目明显在为“长期运行、多平台承载、可持续扩展的个人/团队 Agent”做地基，因此对会话持久化、配置分层、环境后端、多 profile、技能系统都投入很多。

目前还不确定但值得后续验证的点：

- 各工具之间的真实调用频率和最主要瓶颈在哪里。
- memory、skills、session search 三者在真实使用中是否会产生信息污染和提示词竞争。
- gateway 多平台并发下的吞吐瓶颈究竟更多来自模型调用、SQLite 争用，还是外部平台 API 限制。

### 下一步最值得优先深挖什么

如果只选一个方向，我建议下一轮优先深挖：

**为什么这个项目必须做“上下文压缩 + 会话存储 + 记忆/技能分层”，以及这几层如何避免互相污染。**

原因很直接：

- 这是 Agent 项目和普通聊天应用最本质的分水岭。
- 这也是长期运行 Agent 最容易失败的地方。
- Hermes 在这块已经投入了很多明显的工程设计，值得作为理解全局的抓手。

如果你更偏工程架构，也可以把后续深挖顺序排成：

1. 上下文/记忆分层
2. 工具调用与 toolset 边界
3. gateway 多平台会话模型
4. delegate/subagent 机制
