### 1. 路由与 Planner 的边界

本轮只深挖一个问题：为什么 AgenticSeek 不把所有请求都直接分给 `code`、`files`、`web`、`talk` 这些专职 Agent，而是要先在更上游判断“这件事是否已经复杂到必须先做规划”。

这轮只展开 3 个支撑点：

- Planner 解决的到底不是哪一类能力问题，而是哪一类任务结构问题。
- 这条边界在代码里是怎样被触发的。
- 为什么“全部直接路由”太弱，而“全部先进 Planner”又太重。

暂时不展开的内容：

- Planner 的 JSON 计划如何解析和回写。
- 各专职 Agent 的工具执行细节。
- Browser Agent 的网页导航链路。

这些内容当然相关，但它们都发生在“已经决定进入 Planner 之后”。如果先把下游全部展开，反而会模糊这轮真正要理解的判断边界。

先说结论：`Planner` 在这个系统里不是一个和 `web`、`code`、`files` 并列的普通能力 Agent，它更像一个“任务升级层”。专职 Agent 解决的是“谁最适合执行当前动作”，而 Planner 解决的是“这个请求是否已经不是一个单动作，而是一串有依赖关系的动作”。

这两者看起来都叫“路由”，但保护的东西不同。比如“帮我找一下某个 PDF”这种请求，本质上只有一个动作，交给 `FileAgent` 就够了；“先找到 PDF，再读内容，再去网站上回复别人”这种请求，问题已经不再是“找文件”还是“上网页”，而是顺序、依赖和中间结果传递。这里如果还只让路由器选一个专职 Agent，那个 Agent 就得自己隐式承担拆步骤、记依赖、决定先后顺序的工作，而这正是普通专职 Agent 最不擅长、也最容易失控的地方。

所以这个项目把边界划在“复杂度”上，而不是划在“任务领域”上。代码里很清楚地体现了这一点。`Interaction` 在真正执行前，会把用户请求先交给 `AgentRouter`。`AgentRouter` 并不是直接在 `web`、`code`、`files`、`talk` 之间四选一，而是先做三步预处理：

- 只取请求的第一句。
- 把非英文请求翻译成英文。
- 先跑一个单独的复杂度分类器。

只有在复杂度被判为 `LOW` 时，它才会继续进入普通 Agent 选择；一旦复杂度被判为 `HIGH`，或者分类器自己都没把握，系统就直接把请求抬升给 `PlannerAgent`。这说明在作者的心智模型里，Planner 不是“第五种业务能力”，而是一个优先级更高的保护机制。先判断要不要规划，再判断该由哪个专职 Agent 干活。

这里有一个非常值得学习的设计点：这个边界判断不是按领域来的，而是按组合度来的。网页任务不一定进 Planner，代码任务也不一定进 Planner。真正会把请求推向 Planner 的，往往是这几类信号：

- 一个请求同时要求“查资料 + 产出代码/文件/报告”。
- 一个请求需要前一步结果作为后一步输入。
- 一个请求涉及多个环境切换，比如本地文件、网页和代码执行来回跳转。
- 一个请求本身模糊，但明显不是一句话能直接落到单工具上的动作。

这也是为什么训练样例里，“查天气”“写一个简单脚本”会被视作低复杂度，但“找 API 再做应用”“读文件再去网页执行动作”“按 README 安装一个项目”会被视作高复杂度。前者只是要一个执行者，后者需要一个编排者。

如果没有这条边界，会发生什么？最直接的问题是，专职 Agent 会被迫承担本不该由它承担的任务管理责任。`CodeAgent` 本来适合“写代码并根据执行反馈修正”，它不适合决定“我是不是应该先让别人去找 API、先建目录、先读文件”；`BrowserAgent` 适合“搜索、阅读、导航、记笔记”，它也不适合决定“网页上拿到的东西下一步要不要交给写代码的 Agent”。一旦让它们直接面对复合目标，系统就会出现两种常见退化：

- 要么某个 Agent 只完成自己最擅长的一段，整体目标没有闭环。
- 要么某个 Agent 被迫在自己的提示词里“脑补计划”，结果计划不可见、不可校验，也没法把中间结果明确交给下一步。

而如果反过来，把所有请求都先扔进 Planner，也一样会出问题。因为 Planner 的代价并不低。它要先让模型输出符合格式的 JSON 计划，解析失败时还要重试；它还会为每个子任务再启动下游 Agent，把中间结果按 `need` 依赖重新拼回去。对真正复杂的请求，这个成本是值得的；但对“讲个笑话”“找个文件”“写个函数”这种本来一步就能完成的任务来说，这会把一个简单动作膨胀成多轮规划、更多 token 消耗和更高失败面。

也就是说，这条边界本质上是在平衡两种成本：

- 不做规划的成本：复杂任务容易失控，步骤顺序和依赖关系没人兜底。
- 过度规划的成本：简单任务被强行套上计划层，系统更慢、更脆、格式依赖更重。

AgenticSeek 当前阶段选择的方案非常务实：用一个轻量复杂度分类器先做门控，而且在低置信度时宁可保守地升级到 Planner。这不是因为 Planner 更高级，而是因为对于模糊的复合请求，错放给单 Agent 的代价通常比“多走一层 Planner”的代价更大。这个策略尤其适合本地模型场景，因为本地模型的指令稳定性通常弱于闭源大模型，系统需要在上游多做一点保护。

不过，这条边界也不是没有代价。它当前依赖的是“第一句 + 翻译 + few-shot 分类”的轻量判断，而不是对整段请求做真正的任务图分析。这意味着它很便宜，也容易本地跑，但也意味着边界有一定启发式色彩。例如，一个真正复杂的需求如果被用户拆成第二句才说清，路由器可能会低估它；反过来，一个表述很长但其实只有单步目标的请求，也可能被保守地送进 Planner。这里不是实现错误，而是当前阶段很典型的工程取舍：先用便宜、可落地的启发式门控，把大部分明显复杂任务挡进规划层，而不是一开始就做昂贵的全量语义编排。

如果将来项目继续变复杂，这条边界大概率会升级，而升级方向也基本可以预见。最自然的几条路是：

- 不再只看第一句，而是分析整段请求中的多意图和依赖词。
- 不只输出 `HIGH/LOW`，而是输出更细的任务形态，比如“单步执行”“多步串行”“可并行分叉”。
- 让 Planner 不再只是顺序拆分，而是显式构造一个任务 DAG。
- 在路由器和 Planner 之间增加“轻规划”层，只对边界模糊的请求生成最小计划。

所以，`路由与 Planner 的边界` 这个问题真正值得学的地方，不是“作者用了一个分类器”，而是他先把系统里的两类失败分开了：一种失败来自选错执行者，另一种失败来自根本不该直接执行。专职路由处理前者，Planner 处理后者。理解了这一层，就能明白为什么这个项目虽然还不算重型平台，却已经不再是一个简单的多工具聊天壳。

如果再往前推一步，你会发现这个项目为什么不是两层，而是刻意拆成 `Router -> 专用 Agent -> Planner` 三层，本质上是在拆三种不同的不确定性。

第一层 `Router` 处理的是最便宜、也最上游的不确定性：这个请求现在应该直接执行，还是应该先升级成规划问题；如果可以直接执行，那更像是网页、代码、文件还是闲聊。它的职责必须轻，因为它运行在每一个请求前面。如果把 Router 和 Planner 合并，意味着每个请求都要先让一个生成式规划器“想一遍”，那简单请求的平均成本会被明显拉高；如果把 Router 和专用 Agent 合并，等于让每个 Agent 自己先判断“是不是该由我接”，系统就会从一个集中门控退化成多个分散猜测。

第二层 `专用 Agent` 处理的是领域执行不确定性。它们并不是简单的提示词皮肤，而是把“这类任务通常怎样落地”固化进了自己的工具集、重试逻辑和工作约束里。`CodeAgent` 知道自己会遇到代码块、执行反馈和纠错循环；`BrowserAgent` 知道自己会先搜、再看页、再记笔记、再导航；`FileAgent` 知道自己的边界是工作目录和文件动作。也就是说，这一层的价值不是“给模型换个名字”，而是把不同任务类型各自最常见的执行流程封装起来。如果没有专用 Agent，Planner 最终就只能直接面向底层工具，既要决定顺序，又要亲自管浏览器、文件、代码执行细节，提示词会迅速变重，出错面也会大很多。

第三层 `Planner` 处理的则是结构级不确定性。它不关心单步网页搜索怎么做，也不关心单段代码怎么修，而是关心“为了完成这个目标，先做什么、后做什么、谁依赖谁”。这正是前两层都不适合承担的东西。Router 太轻，只适合快速门控；专用 Agent 太专，只适合把一个局部动作做深。只有 Planner 适合把复合目标拆成带 `id` 和 `need` 依赖的步骤，再把步骤下发给真正擅长执行的 Agent。

所以这三层不是功能重复，而是决策粒度不同：

- Router 决定“要不要升级成规划问题”。
- Planner 决定“如果要规划，应该拆成哪些步骤、谁先谁后”。
- 专用 Agent 决定“这个步骤在本领域里怎样真正执行到环境里”。

这也是为什么它们不能轻易两两合并。把 Router 吞进 Planner，会得到一个对所有请求都过度思考的重前门；把 Planner 吞进专用 Agent，会得到一批各自偷偷脑补计划、但计划不可见也不可复用的执行器；把专用 Agent 吞进 Planner，会得到一个提示词极重、同时管理所有工具细节的全能大 Agent，短期看更“统一”，长期看最难稳定。

从工程视角看，这三层实际上是在刻意隔离三种成本：

- Router 隔离的是分类成本，要求便宜、快、保守。
- Planner 隔离的是编排成本，允许更贵，但只在复杂任务上触发。
- 专用 Agent 隔离的是执行成本，让每种动作都在最合适的提示词和工具闭环里完成。

这就是它拆成三层的根本原因。不是为了层次好看，也不是为了喊“多 Agent”这个概念，而是因为单个层次无法同时做到便宜门控、可靠编排和领域执行。把这三件事硬塞给同一层，系统会很快变成要么过重、要么失控、要么两者兼有。
