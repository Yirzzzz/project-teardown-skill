# AgenticSeek 架构拆解

> Skill usage marker: used the project-teardown skill

## 第一部分：当前全局总览

### 一句话判断

AgenticSeek 本质上不是一个“更会聊天的本地 LLM UI”，而是一个面向个人桌面场景的本地代理编排器：它用一个 Python 后端把本地/远程模型、任务路由、专职 Agent、工具执行、网页自动化和前端状态轮询串成了一条可执行任务链。

### 这个项目本质上是什么

- 它面向的不是纯问答场景，而是“让模型替用户动手做事”的场景，比如搜网页、看网页、找本地文件、写代码、执行脚本、拆复杂任务。
- 它的核心不是“模型多强”，而是“模型输出能不能被系统接住并转成动作”。
- 它当前更像一个本地优先的单体 Agent Runtime，而不是一个已经做完隔离、并发调度、持久任务队列、多租户权限控制的 Agent 平台。
- `frontend` 提供 React 界面，`api.py`/`cli.py` 提供 Web 与 CLI 两个入口，`sources/` 是主编排层，`llm_router/` 提供本地分类模型，`llm_server/` 更像兼容不同本地推理后端的辅助层。

### 它解决的核心业务痛点

- 纯聊天式 LLM 无法稳定操作本地文件、浏览器和命令行，所以需要把“生成文本”变成“驱动工具和环境”。
- 面向隐私敏感用户时，云端 Agent 产品的默认模式不成立，所以它强调本地运行、本地工作目录、本地浏览器、本地搜索代理。
- 本地模型能力波动很大，复杂任务如果直接交给单个 Agent，常常会又慢又乱，所以这里引入了路由和规划层。
- 网页任务如果只靠模型“想象自己在浏览”，很容易幻觉，因此项目把搜索、打开页面、记笔记、填表、截图拆成明确环节。

### 一个典型请求如何穿过系统

1. 用户从 React 前端或 CLI 提交请求。
2. `api.py` 在启动时初始化 `Provider`、`Browser`、多种 Agent 和 `Interaction`，后续请求复用这一套运行时对象。
3. `Interaction` 把查询交给 `AgentRouter`。
4. `AgentRouter` 先做语言识别与翻译，再做复杂度判断。
5. 如果任务被判为低复杂度，就直接路由到 `casual`、`code`、`files`、`web` 之一；如果被判为高复杂度，就转给 `PlannerAgent`。
6. 专职 Agent 会把系统提示词和记忆送给 `Provider` 调用模型；如果模型输出了特定 fenced block，运行时就调用对应工具执行。
7. 工具执行结果会被重新塞回 Agent 的记忆里，促使模型继续修正、补充或收束答案。
8. 如果走的是 Planner 路径，Planner 会先让模型产出 JSON 计划，再顺序调用其他 Agent 执行每一步，并在失败后尝试更新剩余计划。
9. 最终答案、推理文本、工具执行块和浏览器截图通过 `/query`、`/latest_answer`、`/screenshots/...` 回到前端；前端通过轮询展示中间状态。

### 最核心的架构思想

这个项目最核心的设计不是“多模型”，而是“让通用 LLM 通过轻量结构化文本协议驱动动作”。

具体说，它没有强依赖某一家模型厂商的 function calling，也没有先做一个很重的分布式 Agent 平台；它采用了更适合本地开源模型和 side-project 阶段的办法：

- 用路由决定谁来做。
- 用 JSON 计划表达复杂任务拆解。
- 用 fenced code/query block 表达工具调用。
- 用执行反馈把真实环境结果重新喂给模型。

这样做的好处是兼容面广、本地可跑、工程门槛不高；代价是很多可靠性建立在提示词格式、解析器和重试循环上，而不是建立在强类型工具协议和严格调度系统上。

### 当前最值得先理解的 4 个机制

#### 1. 路由与复杂度门控

它不是简单地“选哪个 Agent”，而是先判断这是不是一个应该进入规划层的任务。没有这一层，系统会在两个坏结果之间摇摆：要么所有请求都走 Planner，成本高且啰嗦；要么所有请求都直接交给单个 Agent，复杂任务容易失控。

#### 2. Planner 的任务分解与回写

Planner 才是这个项目里“多 Agent”真正落地的地方。它要求模型输出包含 `agent`、`id`、`task`、`need` 的 JSON 计划，再按顺序调专职 Agent 干活，并在某一步失败后尝试改后续计划。没有这一层，这个项目就更像几个平级工具 Agent 的集合，而不是一个能处理复合目标的执行器。

#### 3. 工具调用闭环

代码、文件、搜索都不是模型直接“会做”，而是模型先输出 block，运行时再解析、执行、反馈、重试。这一层是“文本世界”和“真实环境”之间的桥。没有它，模型只能描述动作，不能真正完成动作。

#### 4. 浏览器代理链路

网页任务不是单次搜索就结束，而是“生成查询词 -> 用 SearxNG 搜索 -> 选链接 -> 打开页面 -> 提取文本/表单 -> 记笔记 -> 决定继续还是退出 -> 总结”。这条链路是 AgenticSeek 与普通聊天助手差异最大的能力之一，因为它试图把“查网页”升级成“真的在网页里执行任务”。

### 当前大图总结

#### 已确认

- 后端主心脏在 `sources/`，整体是 Python 单体编排。
- 前端只是展示层，真正的 Agent 状态、推理、工具执行和截图都在后端维护。
- 复杂任务并不是并行子代理系统，而是由 Planner 顺序调度专职 Agent。
- 当前运行模型明显偏单用户、本机工作站模式，`WORK_DIR` 是实际文件操作边界。
- 项目支持本地模型，也支持 OpenAI、DeepSeek、Together、Anthropic 等 API，但其主叙事仍然是本地优先。

#### 合理推断

- `docker-compose.yml` 里的 Redis/Valkey、以及代码里的 Celery 依赖，更像为未来异步化和服务化预留，而不是当前主执行链路的核心。
- `llm_server/` 的定位更偏兼容层或实验层，用于把某些本地推理后端包成统一 HTTP 接口，而不是唯一主路径。
- 这个项目当前最优先的目标是“先把端到端能力跑通”，而不是“先把执行可靠性、并发、安全隔离全部做满”。

#### 暂未知

- 还看不到成熟的多用户隔离设计。
- 还看不到严格的任务级权限系统或强沙箱边界。
- 还看不到真正的持久任务编排、恢复点、审计链路和生产级观测体系。

### 当前阶段为什么会选这种方案

如果现在就做成一个完整的分布式 Agent 平台，工程成本会远高于项目当前阶段的收益，尤其这还是一个强调本地运行、个人使用、模型可替换的项目。

所以它当前选择的是一种非常典型的中间路线：

- 比“一个聊天窗口 + 一个模型”复杂，因为必须真的能做事。
- 又比“多租户、强隔离、可恢复、并行任务平台”简单，因为它优先解决的是个人本地自动化，而不是企业级编排。

这正是这个项目现在最重要的工程取舍。

### 下一步最值得优先深挖的方向

优先建议深挖“路由与 Planner 的边界”。

原因不是它名字听起来高级，而是它决定了这个项目为什么不是一个单 Agent 工具箱：什么任务直接交给专职 Agent，什么任务必须先拆计划，Planner 生成的 JSON 计划为什么能代表“多 Agent 协作”，这些问题一旦看懂，整套系统的大部分设计都会变得顺理成章。
