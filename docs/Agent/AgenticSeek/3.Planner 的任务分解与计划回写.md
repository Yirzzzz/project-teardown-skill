### 2. Planner 的任务分解与计划回写

本轮只解释一个核心问题：Planner 为什么不只是“列一个待办清单”，而是要把一个模糊目标改写成一份系统可执行、可传递、可修正的计划。

这轮只展开 4 个支撑点：

- 为什么计划必须是结构化 JSON，而不是自然语言清单。
- 为什么步骤之间要显式写 `need` 依赖。
- 为什么任务执行过程中还要允许“计划回写”。
- 为什么当前实现选择顺序执行，而不是并行调度。

暂时不展开的内容：

- 各子 Agent 内部怎样执行网页、文件和代码动作。
- 工具 block 的解析和执行闭环。
- 更细的错误处理与局部实现缺陷。

因为这轮真正要理解的不是“某一步是怎么跑的”，而是 Planner 怎样把一个原始目标变成后端能接住的执行结构。

先说本质。Planner 在这里做的不是普通意义上的“思考一下”，而是把用户的目标从自然语言，翻译成一份多 Agent 之间可以交换的任务合同。这个合同至少要回答四件事：

- 这次一共有几步。
- 每一步该由哪个 Agent 做。
- 当前步骤依赖前面哪些结果。
- 当某一步失败后，后续步骤是否还成立。

如果没有这份合同，系统其实仍然停留在“一个模型临时脑补执行顺序”的层面。那样虽然表面上也能说出“先做 A，再做 B”，但这些步骤对运行时是不可见的。运行时无法校验、无法传递依赖、也无法在失败后只修改后半段计划。Planner 的价值，就在于把这种隐式推理显式化。

第一层关键点，是为什么计划必须落成 JSON，而不是一段普通文本。代码里 `PlannerAgent.make_plan()` 会直接要求模型输出 ```json 包裹的 `plan` 结构，然后 `parse_agent_tasks()` 从中抽取 `agent`、`id`、`task`、`need`。这件事看上去很朴素，实质上非常关键，因为它解决的是“模型输出如何变成系统输入”的问题。

如果 Planner 只是输出这样的自然语言：

“先查 API，再建项目，再写代码，最后总结一下。”

人能看懂，但程序没法稳定消费。系统不知道哪一步对应哪个 Agent，不知道第三步是否依赖第二步的产物，也不知道“最后总结”该拿哪些上下文。JSON 的作用不是为了好看，而是为了把计划从“给人看的建议”变成“给运行时读的协议”。这就是为什么 Planner 需要专门的 `json` 工具标签和单独的解析逻辑。

第二层关键点，是 `need` 为什么必须显式存在。`need` 本质上是在给多步任务补一条“数据流”。很多人第一次看多 Agent 设计时，容易只关注“谁做哪一步”，但真正难的往往不是分工，而是前一步的结果怎样被后一步安全使用。

比如“先找 weather API，再创建项目目录，再生成应用代码”这个例子里，写代码这一步并不是只靠用户原始需求就能完成，它还依赖：

- 前面选出来的 API。
- 前面创建好的工作目录或项目结构。

所以 Planner 不只是说“Coder 去写代码”，还要把 `need: ["2", "3"]` 显式写出来。运行时随后通过 `get_work_result_agent()` 把前序 Agent 的结果提出来，再拼进下一步 Agent 的 prompt。换句话说，`need` 的作用不是做流程图装饰，而是在没有复杂共享内存、没有任务数据库、没有 DAG 引擎的前提下，给系统提供一个最小可用的依赖传递机制。

如果没有 `need`，系统会退化成一种很脆的模式：Planner 虽然能拆步骤，但每个后续 Agent 仍然只能看见自己的任务描述，看不见真正需要的上游产物。那时要么 Planner 被迫把所有前文重新硬塞进每一步的 task 文本里，提示词会迅速膨胀；要么后续 Agent 在缺上下文的情况下自行猜测，最终又退回隐式推理。

第三层关键点，是为什么 Planner 还要支持“计划回写”。很多人会下意识觉得，既然已经先做了计划，那执行时照着跑就好了，为什么还要 `update_plan()`。这个直觉只对“所有步骤都稳定成功”的理想情况成立，而真实环境里的 Agent 任务恰恰不是这样。

这个项目的步骤并不是纯内存里的函数调用，而是网页搜索、文件创建、代码执行这类会和真实环境互动的动作。只要动作落到真实环境，前面一步的失败就会直接污染后面几步的成立条件。比如：

- 没搜到可用 API，后面的编码步骤就不该按原计划继续。
- 文件 Agent 没把项目目录准备好，Coder 拿到旧任务继续写，保存路径就会出问题。
- 某一步工具执行失败，后面的总结步骤也不该假装一切完成。

所以 `update_plan()` 的本质不是“让 Planner 再想一次”，而是把 Planner 变成一个轻量的执行中控制器。它会拿到三类信息：原始目标、刚刚完成的那一步实际结果、以及系统对这一步成功/失败的判断。然后它只重写失败点之后的后半段计划，而不改已经发生的历史步骤。这一点非常重要，因为它说明 Planner 不是全局重算，而是在做一种最小修补。

这解决了一个非常现实的问题：计划在生成时是基于预测，而不是基于事实；只有执行之后，系统才真正知道世界长什么样。所以计划回写本质上是在把“预测驱动的计划”纠正为“事实驱动的后续动作”。

第四层关键点，是为什么当前实现选择顺序执行，而不是并行执行。代码里 `PlannerAgent.process()` 是一个很直接的 while 循环：拿当前步骤，取 `need`，调用对应子 Agent，收集结果，再决定是否更新后续计划。它没有任务图调度器，也没有并行 worker 池来跑多个子 Agent。这不是因为作者不知道并行更快，而是因为在当前阶段，顺序执行能显著降低系统复杂度。

顺序执行的好处在这里非常具体：

- `need` 依赖天然按时间顺序满足，不需要额外解决同步问题。
- 文件、代码、浏览器这些动作很多都有副作用，并行执行很容易互相踩环境。
- 计划回写只需要关心“刚刚那一步”对后续的影响，不必处理多个分支同时失败的合并问题。
- 对本地单用户场景来说，先保证计划可解释、可修补，比先做吞吐量更重要。

也就是说，当前的 Planner 不是一个 DAG 调度器，而是一个顺序的、可回写的任务链执行器。这个选择牺牲了并行速度，但换来了更容易理解的状态模型：任何时刻系统只需要回答“当前步骤是什么，它依赖谁，它执行得怎么样，后面要不要改计划”。

从工程取舍上看，这其实很合理。因为这个项目当前面对的主要痛点不是“任务吞吐量太低”，而是“复合目标如何在本地模型和真实工具之间稳定跑通”。在这个阶段，顺序链比并行图更适合。

所以，Planner 的任务分解与计划回写，真正聪明的地方不在于它会“列步骤”，而在于它做了三次关键降维：

- 用 JSON 把模糊计划降维成运行时可解析的协议。
- 用 `need` 把跨步骤依赖降维成最小的数据传递关系。
- 用计划回写把执行中的不确定性降维成“只修后半段”的局部调整。

这三件事加起来，才让 Planner 从一个“会写 todo list 的 LLM”变成一个真正能驱动多 Agent 协作的上层执行器。

如果将来项目继续演进，这一层最自然的升级方向也很清楚：

- `need` 可能会从简单的步骤 ID，升级成更细粒度的产物引用。
- 计划回写可能不再是整段后半部分重写，而是基于任务状态做局部 patch。
- 顺序链可能升级成显式 DAG，把无依赖步骤并行化。
- Planner 结果可能会落到数据库或任务存储中，而不只是停留在内存和 prompt 里。

但在当前阶段，这个实现已经抓住了本质：复杂任务之所以难，不只是因为要拆步骤，而是因为拆完之后还要让步骤之间真的能传数据、能被修正、能继续执行。Planner 在这个项目里正是在解决这个问题。
