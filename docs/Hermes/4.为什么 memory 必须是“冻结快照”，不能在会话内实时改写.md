### 为什么 memory 必须是“冻结快照”，不能在会话内实时改写

#### 本轮只回答一个核心问题

这一轮只讲一个问题：

**为什么 Hermes 宁愿让 memory “现在写入、下次会话生效”，也不愿意在当前会话里实时把 memory 回灌进 system prompt。**

这个问题如果不看清楚，很容易误以为 Hermes 只是“实现上比较保守”。  
实际上，这里保护的是整个 Agent 的稳定性。

#### Hermes 的实际做法是什么

从代码上看，这不是推测，而是明确设计：

- `run_agent.py` 把 system prompt 缓存在 `self._cached_system_prompt`
- 注释明确写了：一个 session 里只构建一次，尽量保持稳定，以最大化 prefix cache 命中
- `tools/memory_tool.py` 则把 memory 分成两份状态
  - 一份是 live state：实时更新、立刻写磁盘
  - 一份是 `_system_prompt_snapshot`：session 开始时冻结下来，当前会话一直使用

也就是说，Hermes 的设计不是“memory 不实时保存”，而是：

**实时保存，但不实时回灌。**

#### 为什么不能实时改写：第一层原因是缓存会被直接打碎

Hermes 很重视多轮会话的 prompt caching。

`agent/prompt_caching.py` 的策略本质上是在复用稳定前缀：  
system prompt 越稳定，多轮对话越能吃到缓存收益。

如果 memory 每次写入后都立刻重建 system prompt，会发生什么？

- system prompt 变了
- 前缀变了
- cache 命中率掉了
- 每轮成本和延迟都更容易上升

这意味着：

“让 memory 立刻生效”

并不是白来的，它的代价是：

“让整个多轮会话越来越像每轮都在重开新前缀”

所以冻结快照首先保护的是经济性和响应稳定性。

#### 第二层原因是：刚写进去的 memory 会被错误抬高优先级

memory 的本意是长期背景。  
但如果你在会话内实时回灌，它的行为就变了。

原本一条新写入的内容可能只是：

- 一个刚发现的偏好
- 一个临时总结
- 一个未来可能有用的经验

可一旦它立刻回到 system prompt，模型就很容易把它当成：

- 当前高优先级背景
- 已经稳定成立的事实
- 后续每一轮都该继续依赖的前提

这就把“为未来准备的信息”错误地提升到了“当前任务调度层”。

Hermes 选择冻结快照，本质上是在维持一个时间尺度边界：

- 当前任务继续用当前会话原有背景
- 新写入的 memory 留给未来 session 生效

#### 第三层原因是：实时回灌会制造“模型和自己刚写下的内容互相强化”

这是 Agent 系统里很典型的自我污染。

如果过程变成这样：

1. 模型在当前任务中形成一个临时判断
2. 它把这个判断写入 memory
3. 下一轮 system prompt 立刻带着它回来
4. 模型又把它当成稳定背景继续引用

那么系统很快就会出现一种危险现象：

**不是事实越来越可靠，而是模型越来越容易相信自己刚刚写下的东西。**

尤其在调试、探索、信息不完整的任务里，这会很糟糕。  
错误结论会被过早“升格”为长期背景，之后更难被推翻。

冻结快照切断了这个闭环：

- 本会话写 memory，是为了未来
- 不是为了当前会话立刻自我放大

#### 第四层原因是：system prompt 语义会不断漂移

长期 Agent 很怕一种问题：  
用户任务没显式变，但模型的“背景世界”在悄悄变化。

如果 memory 会话内实时更新，那么第 3 轮、第 4 轮、第 5 轮看到的 system prompt 都可能不同。  
这样一来，很多行为会变得难以解释：

- 为什么这一轮判断和上一轮不一样
- 为什么工具选择突然偏了
- 为什么某个刚生成的经验好像瞬间变成了系统级规则

Hermes 的 frozen snapshot 相当于在说：

**同一个 session 里的背景世界观尽量固定，memory 的刷新应该发生在 session 边界，而不是 turn 边界。**

这和数据库里的“稳定读视图”其实很像。

#### 那为什么还要实时写磁盘

因为 Hermes 不是不要即时持久化，而是不要即时激活。

这里其实分成两个动作：

1. `durability`：现在立刻落盘，避免成果丢失
2. `activation`：下次会话再进入 system prompt

这个设计很成熟，因为它同时满足了两件互相冲突的事：

- 记忆不能丢
- 当前会话不能乱

如果等会话结束才写磁盘，durability 不够好。  
如果写完立刻回灌，context hygiene 又会被破坏。

Hermes 选的是中间那条更工程化的路。

#### 为什么这不是小优化，而是 session 稳定性的核心

如果 memory 实时回灌，系统最容易出现三种坏味道：

1. 越聊越慢：因为 system prompt 一直变，缓存收益越来越差
2. 越聊越乱：因为长期背景和当前任务互相抢优先级
3. 越聊越自信：因为模型不断吃到自己刚制造出来的“伪长期事实”

Hermes 的冻结快照，实际上就是在提前阻断这三件事。

所以这个设计真正保护的不是某个文件格式，而是三种系统性质：

- **稳定性**：一个 session 内的背景尽量不漂移
- **经济性**：prefix cache 尽量持续生效
- **认知卫生**：模型不要反复把自己刚写下的内容当作新的系统事实

#### 为什么这很值得学

很多 Agent 项目在 memory 这层最容易犯的错是：

“既然记住了，就立刻喂回去，模型不是会更聪明吗？”

Hermes 的回答更像一个做长期系统的人：

“短期可能更灵一点，但长期会更脏、更贵、更不稳。”

这就是为什么它宁愿牺牲一点即时性，也要保住 session 级别的稳定边界。
