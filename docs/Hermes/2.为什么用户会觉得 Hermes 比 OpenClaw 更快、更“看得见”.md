### 为什么用户会觉得 Hermes 比 OpenClaw 更快、更“看得见”

#### 本轮只回答一个核心问题

这一轮不讨论所有架构细节，只讨论一个问题：

**为什么用户会主观感受到 Hermes 更会干活、更快，而且执行过程更透明。**

这个问题最多只展开 4 个支撑点：

1. 为什么“更快”不只是模型推理速度
2. Hermes 如何减少完成任务所需的总轮数
3. Hermes 如何缩短单轮工具执行耗时
4. Hermes 如何把隐藏等待变成可见进度

暂时不展开的内容：

- memory / skills 长期污染问题
- gateway 多平台并发瓶颈
- prompt caching 的经济性影响

这些都相关，但会把这一轮的主问题冲散。这里先只抓“用户为什么会明显感觉快”。

#### 先把“快”分成两种

用户口中的“快”，在 Agent 系统里其实经常是两种东西叠在一起：

- **真实吞吐更快**：同一个任务，系统更少轮数完成，或者每轮工具执行更快。
- **感知速度更快**：哪怕总耗时接近，但系统持续给你反馈，让你知道它没卡死、正在做什么。

Hermes 之所以容易被感知为更快，不是因为只优化了其中一个，而是这两件事一起做了。

#### 第一层：它先减少“完成任务需要多少轮模型来回”

很多 Agent 系统慢，不是单次 API 很慢，而是模型需要一轮轮地想、调工具、读结果、再想、再调工具。  
只要轮数变多，延迟就会被不断累加。

Hermes 明显在压这个问题，最典型的是 `execute_code`。

`tools/code_execution_tool.py` 不是普通“跑一段 Python”这么简单，它本质上是在做一件很关键的事：

**把原本需要模型亲自发起的一串工具调用，折叠进一个脚本执行回合里。**

脚本可以通过 RPC 调 Hermes 的工具，父进程负责真正分发；而中间那些工具结果默认不会一条条塞回模型上下文。  
这意味着什么？

- 如果任务是“搜 10 个页面，再筛，再抽取，再汇总”，模型不必亲自经历 10 次以上来回。
- 处理逻辑可以在脚本里完成，模型只看最终 stdout。
- 大量中间结果不会把上下文变肥，后续轮次也更轻。

`run_agent.py` 里甚至对纯 `execute_code` 回合做了 iteration refund，这已经非常明确地说明设计者把它视为一种“低成本的程序化工具编排”，而不是普通工具。

如果你从架构角度理解，这不是一个小优化，而是把：

“模型逐步指挥工具”

改成：

“模型把流程编译成一个小程序，再让程序去调工具”

这会直接改变复杂任务的完成速度。

#### 第二层：它会在单轮内部并行，而不是默认所有工具串行

即使不使用 `execute_code`，Hermes 也没有把所有工具调用都老老实实串行执行。

`run_agent.py` 里可以看到三种划分：

- 明确绝不并行的工具
- 明确只读、可并行的工具
- 按路径隔离后可并行的文件工具

而且它设置了 `_MAX_TOOL_WORKERS = 8`。  
这说明 Hermes 对“同一轮多个工具调用”不是 naïve 地一个个跑，而是会判断是否可并发。

这层设计解决的是一个很实际的问题：

如果模型一轮里发出多个读文件、搜文件、会话搜索、网页抽取这类互不影响的工具调用，串行执行会把总等待时间线性叠加。  
而并行执行会把总时间更接近最长的那一个。

这类优化用户通常不会直接说出来，但会直接体现在一句话上：

“它做事更利索。”

因为很多复杂任务真正花时间的部分，不在模型生成，而在 I/O。

#### 第三层：它把上下文负担压低，所以后半程不容易越跑越慢

另一个常见问题是：Agent 前几轮很快，后几轮越来越慢。  
原因不是模型变笨，而是上下文越来越胖，尤其工具结果特别容易堆积。

Hermes 在 `agent/context_compressor.py` 里先做旧 tool result 裁剪，再做中段摘要，而且强调 summary 只是“reference only”。  
这说明它不是简单粗暴删消息，而是在努力维持一个还能持续工作的上下文。

这层设计的价值不是只为了“别爆 token”，更是为了防止系统出现这种体验：

- 前面像个能干的 agent
- 后面逐渐变成一个每轮都很慢、每轮都在背大包袱的聊天模型

如果一个 Agent 想表现出“持续干活的速度感”，这层非常关键。  
因为用户感受到的不是第一轮快不快，而是“做到第六步时有没有明显拖垮”。

#### 第四层：它把隐藏在内部的执行状态显式暴露出来

这是用户“能看到它在干什么”的直接来源。

Hermes 不只是执行工具，还专门做了一整层显示系统：

- `agent/display.py` 里有 tool preview、emoji、spinner、diff、`print_above`
- `run_agent.py` 有 `tool_progress_callback`、`tool.started`、`tool.completed`
- `delegate_tool.py` 会把子代理的工具调用和思考摘要回传给父代理显示

这背后的本质不是 UI 好看，而是：

**把原本用户不可见的 agent 内部状态，转译成连续可见的执行轨迹。**

这会极大改变主观体验。

因为对用户来说，最糟糕的不是“等 12 秒”，而是：

“12 秒里什么都看不到，不知道它死了、卡了、还是在认真工作。”

Hermes 的做法相当于持续告诉用户：

- 现在在读哪个文件
- 现在在搜什么
- 现在子代理在做哪几个任务
- 哪个任务完成了，还剩几个

一旦等待被切成一段段可解释的小动作，用户就会把系统理解为“在执行”，而不是“在发呆”。

#### 为什么这会让人觉得它“比 OpenClaw 更会干活”

因为“会干活”的主观评价，往往不是来自回答更华丽，而是来自这三个信号：

1. 它没有让我重复等很多轮
2. 它每轮动作很实，不拖泥带水
3. 它做事时我能看见过程，而不是只能赌结果

Hermes 恰好分别用三种机制去打这三个点：

- `execute_code` / `delegate_task`：减少大模型亲自调度的轮数
- 单轮工具并行：缩短同一轮的执行时间
- progress callback + display：让执行过程可见

所以从架构角度说，用户说“它更快”，并不只是夸某个模型供应商或参数配置。  
他们其实是在为 Hermes 的 **Agent runtime 设计** 买单。

#### 如果没有这些设计，会发生什么

如果不用这套思路，Agent 往往会落到一种非常典型的慢法：

- 所有工具都串行
- 每一步都让主模型亲自指挥
- 中间结果全部塞进上下文
- 执行过程中几乎没有进度反馈

这样即使模型本身 token 输出速度不慢，用户仍然会觉得：

- 它干活磨蹭
- 它不像在执行，更像在犹豫
- 它一复杂起来就明显变慢

也就是说，慢感很多时候是系统设计造成的，不是模型速度本身造成的。

#### 这一层未来如果再升级，会往哪里走

如果 Hermes 继续沿这条路往前走，后续最自然的升级方向会是：

- 更激进的计划级并行，不只在工具层并行，还能把任务图显式化
- 更细粒度的进度语义，不只是“tool started/completed”，而是阶段性里程碑
- 更强的中间结果折叠，把“可见”与“少占上下文”同时做到极致
- 更系统的观测层，把速度感从 UI 体验扩展成真正可量化的 profiling

这说明它现在的快，不像是偶然调参得来的，而是已经具备一条比较清晰的工程演进路线。
