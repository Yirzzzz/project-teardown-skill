### 上下文压缩、记忆、技能、历史检索之间如何分层

#### 本轮只回答一个核心问题

这一轮只讲一件事：

**为什么 Hermes 不能把“过去的一切”都直接塞进当前上下文，而必须拆成上下文压缩、会话存储、记忆、技能四层。**

本轮最多只展开 4 个支撑点：

1. 如果不分层，Agent 会先死在哪些地方
2. 这四层各自到底在保存什么
3. 它们的边界为什么能减少相互污染
4. Hermes 已经用了哪些具体机制来防污染

暂时不展开：

- 更细的 prompt caching 算法
- 外部 memory provider 的插件生态
- 未来是否要引入向量检索或更复杂的长期记忆系统

这些都相关，但这一轮的主问题是“为什么必须分层”。

#### 先把问题说透：为什么不能只做一个“大记忆上下文”

很多人刚接触 Agent 时会有一个很自然的想法：

“既然模型会忘，那就把历史、经验、用户偏好、工具结果、解决方案都一直放进提示词里不就行了？”

这个思路在短对话里看上去成立，但一旦系统开始长期运行，就会同时撞上四个问题：

1. **上下文会无限膨胀**  
   工具结果、网页内容、文件内容、聊天历史都会涨，模型每一轮都要背更多东西。

2. **不同类型的信息价值周期完全不同**  
   某次 `read_file` 的输出可能只在接下来的 2 轮有用；用户长期偏好可能几个月后还有效；一个调试流程更适合沉淀成技能，而不是一直当会话噪音带着走。

3. **不同类型的信息可信度和作用方式不同**  
   原始历史记录适合做“事实回放”；记忆适合放稳定事实；技能适合放可复用流程；临时上下文只是当前任务的工作集。把它们混在一起，模型会越来越难分清哪些是“背景”，哪些是“当前任务”。

4. **缓存和成本会被打烂**  
   Hermes 明显很在意 prompt cache 稳定性。只要把中间状态不停塞回 system prompt 或前缀，整个会话成本和延迟都会迅速上升。

所以这不是“设计得优雅一点”的问题，而是：

**如果不分层，长期运行的 Agent 会同时出现变慢、变脏、变乱、变贵四种症状。**

#### 这四层各自到底在保存什么

要理解 Hermes 的分层，关键不是记住模块名，而是理解每层保存的是哪一种“时间尺度”的信息。

##### 第一层：当前上下文

这是模型正在看的工作记忆，包含当前会话里仍然活跃的消息、最近工具结果、系统提示、项目上下文等。

它服务的是：

- 当前问题
- 当前任务步骤
- 当前这几轮还在使用的证据

它本质上是短期工作集，不是真正的长期存储。

##### 第二层：会话存储

`hermes_state.py` 把完整 session 存进 SQLite + FTS5。  
这一层保存的不是“该一直注入模型的知识”，而是“以后需要时可以回放和检索的历史事实”。

这非常重要，因为大量过去信息其实不应该常驻上下文，但你又不能丢。  
所以 Hermes 的选择不是“注入”它，而是“存档后按需召回”。

##### 第三层：记忆

`memory_tool.py` 很明确：记忆只该存 durable facts，例如用户偏好、环境事实、长期约定、工具 quirks。  
`prompt_builder.py` 里也直接写了：不要把任务进度、会话结果、临时 TODO 写进 memory；那类东西应交给 `session_search`。

这说明 Hermes 对 memory 的定义不是“任何过去都叫记忆”，而是：

**只有会持续减少未来用户重复纠正成本的事实，才值得进入记忆。**

##### 第四层：技能

技能不是“发生过什么”，而是“以后遇到类似问题应该怎么做”。  
它承载的是流程、套路、工作流、经验固化。

也就是说：

- 会话存储保存“那次做了什么”
- 记忆保存“长期稳定事实是什么”
- 技能保存“以后遇到这类问题该怎么做”

如果你把这三种都叫 memory，系统最终就会变成一锅粥。

#### 这几层为什么刚好互相补位

Hermes 的分层其实是在解决一个核心张力：

**既要保留过去，又不能让过去污染当前。**

它的做法可以概括成一句话：

- 不常用但不能丢的，进会话库
- 稳定且高价值的，进记忆
- 可迁移的方法论，进技能
- 只对当前任务有用的，留在当前上下文，超长后再压缩

这四层不是竞争关系，而是各自回答不同问题：

- 上下文回答：我现在正在干什么
- 会话存储回答：以前到底发生过什么
- 记忆回答：有哪些长期稳定事实以后还要继续用
- 技能回答：有哪些做法值得复用

#### Hermes 是怎么避免它们互相污染的

下面这部分最关键，因为它不是停留在概念上，而是已经落到了代码边界里。

##### 1. 记忆用“冻结快照”，避免会话中途不断改前缀

`tools/memory_tool.py` 的设计非常明确：

- session start 时从磁盘加载
- 生成 `_system_prompt_snapshot`
- 中途写 memory 只更新磁盘和 live state
- **不会回写当前会话的 system prompt**

这层机制的本质是：

**把记忆从“会话内实时漂移的上下文”变成“会话边界上刷新的一组稳定背景”。**

如果不这么做，会出现两个问题：

1. prompt prefix 一直变，缓存价值消失
2. 模型会把刚写进去的记忆当成当前任务高优先级指令，造成自我干扰

Hermes 选择“本会话看冻结快照，下会话再刷新”，其实是在牺牲一点即时性，换稳定性和可控性。

##### 2. `session_search` 走“按需召回 + 摘要”，而不是把所有历史常驻

`tools/session_search_tool.py` 的注释写得很直白：  
先 FTS5 搜索，再用便宜模型总结，返回 focused summaries，而不是把原始 transcript 大段灌回主模型。

这在架构上非常关键，因为它把历史从：

“永远在上下文里”

变成：

“只有相关时才被拉回来，而且还是经过压缩后的相关片段”

这解决了两个污染问题：

- 老历史不会持续占据窗口
- 当前任务只会看到和当前问题相关的过去，而不是整段旧对话噪音

如果 session_search 直接回放原始 transcript，主上下文很快就会被旧会话淹没。

##### 3. 技能不等于历史，也不等于记忆，而是被当成“可主动加载的工作说明书”

Hermes 对技能的处理也很有边界感。

一方面，`build_skills_system_prompt()` 只把技能索引放进 system prompt，告诉模型“有哪些可用技能”；  
另一方面，真正的技能内容是在调用时通过 `skill_view` 或 slash command 加载。

更关键的是，`agent/skill_commands.py` 把 skill invocation 组织成用户消息载荷，而不是每次都重建整个系统提示。  
这说明技能被设计成：

**按任务激活的外部工作说明书，而不是永久驻留的内在人格。**

这样做有两个防污染效果：

1. 不会让所有技能内容常驻上下文
2. 技能只在当前任务相关时才进入工作集，避免“全局 workflow 噪音”

##### 4. 上下文压缩负责清理“当前工作集”，而不是承担长期记忆职能

`ContextCompressor` 的职责并不是“帮系统记住过去”，而是“别让当前工作集失控”。  
它先裁掉旧工具输出，再对中段做 handoff-style summary，并反复强调 summary 只是背景参考，不是当前活跃指令。

这点很重要，因为压缩层如果承担长期记忆职责，就会出问题：

- 它压出来的 summary 很容易混入过期任务
- 它本质上是为当前窗口续命，不适合当长期真相来源

Hermes 相当于明确了：

- 压缩层负责“当前还能跑下去”
- 会话库负责“过去还留得住”
- memory/skills 负责“什么值得跨会话升格”

这就是职责分离。

#### 为什么 Hermes 还专门写了很多“不要混”的提示

这不是文案，而是系统层面的防线。

`agent/prompt_builder.py` 里有几句非常关键的话：

- 不要把 task progress、session outcomes、temporary TODO 存进 memory
- 这些应该用 `session_search` 回忆
- 如果发现了可复用方法，存成 skill
- memory 描述的是用户和长期事实，不是当前运行系统状态

这些提示的真正意义是：

**在模型最容易偷懒的时候，强制它把信息送进正确的层。**

否则模型最自然的行为往往是：

- 觉得有用就都塞进 memory
- 觉得以后可能还用到就都长期保留
- 结果 memory 变成杂物间，skills 变成过期笔记，session_search 反而没人用

Hermes 显然已经意识到这个风险，所以不仅靠存储结构分层，还靠 prompt 约束模型“把东西放对地方”。

#### 如果不这样分层，最容易出现什么污染

可以把典型污染理解成三种：

##### 1. 历史污染当前

旧任务、旧结论、旧文件状态一直残留在上下文里，模型会误以为它们仍然活跃。

##### 2. 临时信息污染长期记忆

一次性的调试日志、任务进度、临时路径被写进 memory，后面每轮都看到，噪音越来越大。

##### 3. 经验流程污染事实层

某次成功的操作步骤本该沉淀成 skill，却被写成 memory 事实；以后模型既学不到完整流程，又反复被模糊表述干扰。

Hermes 的分层，本质上就是在防这三种污染。

#### 为什么这套分层很像“分级存储”，而不是简单的 prompt 工程

如果你从系统角度看，它其实像一种信息生命周期管理：

- 热数据：当前上下文
- 温数据：session_search 可召回的历史
- 冷但高价值的数据：memory
- 可复用操作系统式知识：skills

所以这不是单纯 prompt engineering，而是 Agent 系统里的“分级存储架构”。

一旦你这样理解，就会发现 Hermes 做的不是“我多加了几个工具”，而是：

**给不同寿命、不同用途、不同可信度的信息，安排了不同的容器。**

这正是长期 Agent 和普通聊天工具的本质差异。

#### 这一层未来会往哪里升级

如果 Hermes 继续演进，这一层未来最自然的升级方向是：

- 更强的记忆准入机制，决定什么事实值得升格为 memory
- 更自动化的 skill 提炼，把高质量 session 片段转成工作流
- 更细的 recall routing，自动判断应该查 session、memory 还是 skills
- 更严格的污染检测，例如过期 skill、冲突 memory、重复事实合并

但即使不看未来，当前代码已经很清楚地表达了一个判断：

**长期 Agent 的关键不是“记住更多”，而是“把不同类型的过去放到不同层，并且在需要时以正确形式拿回来”。**
