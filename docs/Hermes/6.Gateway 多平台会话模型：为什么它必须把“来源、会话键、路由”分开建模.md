### Gateway 多平台会话模型：为什么它必须把“来源、会话键、路由”分开建模

#### 本轮只回答一个核心问题

这一轮只讲一个问题：

**为什么 gateway 不能把 Telegram、Discord、Slack、Signal 这些平台都当成“发来一段文本的同一种入口”，而必须专门建一套多平台会话模型。**

如果你只从“把消息送进 agent”这个角度看，gateway 好像只是个转发层。  
但从代码看，Hermes 把它设计得更像一个“多平台会话操作系统”。

#### 先说结论

Gateway 这一层真正要解决的不是“接入平台 API”，而是：

**在不同平台、不同聊天结构、不同用户隔离策略下，稳定地回答这三个问题：**

1. 这条消息到底来自哪里
2. 它应该并入哪条会话
3. 结果将来应该回到哪里

这三个问题如果混在一起，系统很快就会乱。

#### 为什么“多平台”真正难的不是适配器，而是会话身份

很多人直觉上会把多平台理解为：

- Telegram 一个 adapter
- Discord 一个 adapter
- Slack 一个 adapter

接好 API 就完了。

但 Agent 系统真正难的是：  
**不同平台上的“同一段对话”到底应该怎样被定义。**

例如：

- DM 是不是天然一人一会话
- 群聊里是不是每个人都应该有独立上下文
- 线程里是不是应该共享同一条会话
- forum topic / Discord thread / Slack thread 算不算新的会话边界
- `/stop`、中断、排队、恢复，到底应该作用在哪个粒度

这些都不是 adapter 层能单独解决的。  
所以 Hermes 在 `gateway/session.py` 里单独抽了会话模型。

#### 第一层：`SessionSource` 解决“消息从哪里来”

Hermes 没有把来源简化成一个 `chat_id` 字符串，而是建了 `SessionSource`：

- `platform`
- `chat_id`
- `chat_type`
- `user_id`
- `thread_id`
- `chat_topic`
- 以及某些平台的替代 ID

这说明 Hermes 的判断是：

**对 Agent 来说，“来源”不是单一主键，而是一组结构化上下文。**

为什么要这么做？

因为“同一个 chat_id”在不同平台和不同聊天形态里的含义并不一样。  
只有把来源结构化，后面才有可能正确决定会话隔离和回传路径。

#### 第二层：`build_session_key()` 解决“这条消息属于哪条会话”

这是 gateway 多平台模型最核心的一层。

`build_session_key()` 不是简单拼个平台名，而是在显式编码会话边界规则。

从代码能看出几条很重要的规则：

- DM 默认按私聊隔离
- group/channel 默认以父 chat 为主键
- 群聊里是否按用户隔离，由 `group_sessions_per_user` 控制
- thread 默认共享，不按用户继续拆，除非显式开启 `thread_sessions_per_user`

这个设计非常有意思，因为它不是追求“越细越安全”，而是在追求：

**会话隔离粒度要符合用户直觉。**

例如在线程里，默认共享会话是合理的，因为线程本来就是围绕同一主题展开；  
如果每个参与者都在同一 thread 里获得一条独立 session，用户体验反而会变得怪异。

所以 `build_session_key()` 本质上是在做一件很关键的事：

**把“聊天平台的 UI 结构”翻译成“Agent 应该如何理解会话边界”。**

#### 为什么不能直接用 `chat_id` 当 session key

因为那样会立刻出现两类错误。

第一类是隔离不够：

- 群聊里所有人共用一个上下文
- 不同用户的偏好、任务、上下文互相污染

第二类是隔离过头：

- 本来应该共享的 thread 被拆成多个 session
- 用户以为大家在接着同一条线说话，agent 却像失忆一样看不到别人的上下文

Hermes 用配置项：

- `group_sessions_per_user`
- `thread_sessions_per_user`

去控制这件事，本质上是在承认：

**没有一个“永远正确”的全局会话粒度，必须按聊天形态显式建模。**

#### 第三层：`SessionContext` 解决“Agent 在当前平台语境里应该知道什么”

Hermes 不是只把 session key 算出来就完了，还会构建 `SessionContext`，再通过 `build_session_context_prompt()` 注入给 agent。

这里注入的不是历史内容，而是运行语境：

- 当前消息来自哪个平台
- 是 DM、group 还是 thread
- 当前连接了哪些平台
- 有哪些 home channel
- 定时任务或结果可以投递到哪里

这层的价值很大，因为多平台 Agent 最容易犯的一类错是：

**模型不知道自己现在在什么语境里讲话。**

比如它可能误以为自己能搜 Slack 历史、能提及 Discord 全员、能把结果发到某个平台，但实际上 adapter 不支持。  
Hermes 用 platform-specific notes 显式把这些边界告诉模型。

所以 `SessionContext` 不是为了“显示更多元数据”，而是为了减少平台幻觉。

#### 第四层：`DeliveryRouter` 解决“结果应该回到哪里”

多平台系统里，“来源”和“目的地”不能混为一谈。

你收到一条消息的地方，不一定就是未来结果的唯一归宿。  
Hermes 的 `DeliveryRouter` 和 `DeliveryTarget` 明确支持：

- `origin`
- `local`
- 某个平台 home channel
- `platform:chat_id`
- 甚至 thread 级别目标

这说明 gateway 不只是对话入口，也是一个跨平台投递层。

这点非常重要，因为一旦 Agent 支持 cron、background task、跨平台消息发送，系统就必须把：

- 会话来源
- 默认投递目标
- 显式投递目标

三者拆开，否则路由一定会乱。

#### 为什么 reset policy 也必须挂在这个模型上

`SessionStore` 不只是做 key 到 session_id 的映射，它还管理 reset policy：

- idle reset
- daily reset
- suspended session
- active background processes 时不 reset

这说明 Hermes 认为“会话”不是一个静态 ID，而是一个有生命周期的实体。

这很关键，因为多平台对话不是 CLI 那种单窗口同步流程。  
消息平台天然会遇到：

- 昨天说了一半，今天再来
- 这个群很久没动了，要不要清上下文
- 正在跑后台任务时，能不能强制 reset
- gateway 重启后，要不要继续原会话，还是强制 fresh start

所以 reset policy 必须绑定在 session model 上，而不是散落在 adapter 里。

否则你会出现一种典型错误：

- 平台知道消息来了
- Agent 知道自己要继续
- 但系统并不知道“这条继续”在会话生命周期上是否仍然合理

#### 为什么 interrupt / queue 也要用 `session_key`，而不能只认 `chat_id`

`gateway/run.py` 里有很多地方都强调：

- pending message 用 `session_key`
- interrupt 检查用 `session_key`
- running agent 跟踪也用 `session_key`

这是因为在多平台甚至同平台复杂结构里，`chat_id` 不足以表达一个活跃对话单元。

如果只用 `chat_id`：

- 同一个群里的不同用户会互相打断
- 同一个 thread 和主频道可能互相串扰
- queue/pending message 可能被送进错误上下文

Hermes 明显已经踩过这类坑，所以它把中断、排队、恢复、缓存 agent 实例，全都绑在 session key 上。

换句话说：

**session key 才是 gateway 世界里的“真正对话单位”，不是 platform，也不是 chat_id。**

#### 为什么 gateway 还要缓存 AIAgent 实例

`gateway/run.py` 里有一段很关键的注释：

- 按 `session_key` 缓存 `AIAgent`
- 否则每条消息都新建 agent，会重建 system prompt
- 这会破坏 prefix cache，成本和延迟都会上升

这一点说明 gateway 的会话模型不只是为了正确性，也是为了性能。

也就是说，session key 一旦算对了，它不仅决定上下文归属，还决定：

- 哪个缓存 agent 应该被复用
- 哪条会话能延续 prompt cache
- 哪个中断该打到哪个执行中的 agent

所以这其实是“正确性 + 性能 + UX”三合一的主键设计。

#### 这套模型真正解决了什么痛点

如果没有这套多平台会话模型，系统最容易出现四种灾难：

1. **上下文串台**  
   群聊用户 A 的任务污染用户 B，或者 thread 和主频道互相串话。

2. **投递错位**  
   结果回到错误的 chat、错误的平台、错误的 thread。

3. **中断错位**  
   新消息打断了不该打断的 agent，或者 pending message 被送进错误会话。

4. **生命周期混乱**  
   某条会话该 reset 时没 reset，不该 reset 时却被清空。

Hermes 的 gateway 模型，本质上是在防这四种灾难。

#### 为什么这层值得学

很多人做多平台 Agent 时，注意力容易全放在“怎么接 Telegram / Discord API”。  
Hermes 更值得学的地方反而是：

**一旦平台接进来了，如何定义“对话单位”本身。**

它的核心不是 adapter，而是这套判断：

- 什么叫同一条会话
- 什么叫共享上下文
- 什么叫应该隔离
- 什么叫应该回到 origin

只有这层先定对，多平台 Agent 才不会长成一堆彼此打架的消息转发器。
