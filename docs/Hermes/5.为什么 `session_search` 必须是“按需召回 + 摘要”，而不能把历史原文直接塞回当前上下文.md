### 7. 为什么 `session_search` 必须是“按需召回 + 摘要”，而不能把历史原文直接塞回当前上下文

#### 本轮只回答一个核心问题

这一轮只讲一个问题：

**为什么 Hermes 不把过去会话长期挂在当前上下文里，而是先存到 SQLite，再在需要时检索，再把结果摘要后返回。**

这个设计看上去多了一层检索和摘要，像是“绕路”。  
但其实这层绕路，恰恰是在保护主上下文。

#### 先说结论

`session_search` 的本质不是“帮模型记住过去”，而是：

**把“过去发生过什么”从主上下文里剥离出去，变成一个需要时再调用的外部事实库。**

Hermes 的主张不是“让历史一直跟着模型”，而是：

“历史是档案，不是常驻人格。”

#### 如果把历史原文直接塞回当前上下文，会先出什么问题

最先爆炸的不是准确率，而是上下文质量。

因为过去会话的原文天然有四个特点：

1. 很长  
   一次完整 session 往往远比几条 memory 大得多。

2. 很杂  
   里面混有用户问题、工具结果、错误尝试、半成品结论、已经过期的路径和状态。

3. 很旧  
   过去成立的上下文，不代表现在还成立。

4. 很像“当前任务仍在进行”  
   原始 transcript 的语气和结构，对模型来说太像正在发生的对话，容易被误判为仍然活跃。

如果你把这些东西直接塞回主上下文，就会产生一种非常典型的污染：

- 当前任务只需要一句过去结论
- 结果你把整段旧过程、旧尝试、旧失败都一起带回来了

主模型就会开始在“当前任务”和“历史残影”之间分神。

#### Hermes 的做法为什么更像检索系统，而不是长上下文聊天

`tools/session_search_tool.py` 的流程写得很清楚：

1. 先用 SQLite FTS5 搜索相关消息
2. 选出最相关的 session
3. 加载这些 session 的对话
4. 对原文做截断和聚焦
5. 再让便宜模型做 focused summary
6. 返回的是总结结果，而不是整段 transcript

这说明 `session_search` 在架构上的定位不是：

“让主模型重新读一遍历史”

而是：

“让一个专门的检索-压缩流程替主模型消化历史，再把结论喂回来”

这和 RAG 的思路很像，但目标更具体：

不是召回知识文档，而是召回“过去的 agent 行为记录”。

#### 为什么必须先检索，而不是默认全量带历史

因为过去会话的绝大多数内容，对当前问题都是无关的。

Hermes 用 FTS5 的意义，不只是搜索更快，而是先做一次价值筛选：

- 哪些 session 和当前问题相关
- 哪些是 top N
- 哪些甚至根本不值得看

这一步很关键，因为如果不检索筛选，后面的摘要也是在大垃圾堆里做二次压缩。  
你会得到一份“压缩过的噪音”，而不是相关历史。

所以 `session_search` 的第一道防线不是摘要，而是相关性筛选。

#### 为什么检索到之后还不能直接把原文交给主模型

因为“相关”不等于“适合直接注入”。

即使是最相关的一次旧会话，里面也常常包含：

- 中途试错
- 已经被否定的方向
- 很长的工具输出
- 当时环境下才成立的状态

如果你把原文整段交给主模型，它很可能会遇到两个问题：

1. **吃太多 token**  
   当前任务还没推进，窗口先被旧历史占掉一大块。

2. **误把过程当结论**  
   历史里曾经出现过的尝试、怀疑、临时判断，会和真正结果混在一起。

Hermes 所以才加了一层摘要，而且摘要 prompt 明确要求聚焦：

- 用户当时想完成什么
- 做了哪些动作
- 得到了什么结果
- 关键命令、文件、URL、技术细节是什么
- 有什么未解决项

这意味着 Hermes 不是想“再现旧对话”，而是想“提炼旧对话对当前有用的结论”。

#### 为什么这比原文回放更不容易污染当前任务

因为摘要天然把历史降了一层优先级。

原文 transcript 对模型来说像是一段“另一条仍然活着的对话流”。  
摘要则更像：

“这是关于过去的一份结构化回顾。”

这个语义差别非常重要。

一旦你把旧会话变成“总结”，模型更容易把它当作背景证据，而不是当作当前会话的延长线。  
这会显著减少一种常见污染：

**把过去未完成的事情误认为现在仍该继续做。**

#### 为什么 Hermes 还要排除当前 session

`session_search_tool.py` 里明确说了：当前 session 会被排除，因为 agent 已经有当前上下文。

这看起来像小细节，但其实非常关键。

如果当前 session 也允许被 `session_search` 召回，就会出现一种重复注入：

- 当前会话已经在上下文里
- 再从数据库查回来一次
- 还要再摘要一遍

这样不仅浪费 token，还会制造“同一份信息以两种形式重复出现”的干扰。  
Hermes 这里等于是在防“同源信息双重注入”。

#### 为什么它要用便宜模型做摘要，而不是让主模型自己总结

这背后的思路很现实：

- 主模型的上下文很宝贵
- 主模型的推理预算也宝贵
- 检索历史这件事本身更像预处理工作

所以 Hermes 把这类工作交给 auxiliary model。  
这是一种典型的 agent 分工：

- 主模型负责当前任务决策
- 辅助模型负责把外部历史压缩成可消费形式

这样做有两个好处：

1. 主上下文更干净
2. 成本更低

也就是说，`session_search` 不是单纯“多了个工具”，而是把“历史消化”这件事外包了。

#### 如果没有这层设计，最常见的坏味道是什么

如果系统只是简单把旧 transcript 回放给主模型，最容易出现三种坏味道：

1. **历史挤占当前任务窗口**  
   用户问了新问题，模型却先花很多 token 重新读旧事。

2. **旧过程污染当前判断**  
   过去试错路径、过期状态、临时结论跑进来，影响当前决策。

3. **模型越来越像在和“过去的自己”聊天**  
   当前会话不再专注于当前任务，而是在不停和旧轨迹纠缠。

Hermes 的检索 + 摘要，本质上是在阻断这三件事。

#### 这和 memory 的关系是什么

这里最容易学到的一点是：

- memory 负责稳定事实
- `session_search` 负责具体历史

如果一件事只是“那次我们怎么做过”，它更适合进入 session archive，然后按需 search。  
只有当它真的稳定、值得长期减少用户重复纠正成本时，才应该升格到 memory。

也就是说，`session_search` 其实是在给 memory 减压。

没有 `session_search`，模型最容易偷懒的做法就是：

“反正以后可能还要用，那我全写进 memory 吧。”

这会直接把 memory 变成垃圾场。

#### 所以这个设计真正保护的是什么

它保护的是主上下文的“工作记忆纯度”。

Hermes 显然在坚持一个原则：

**当前上下文应该主要承载当前任务，而不是承担完整历史档案库的职责。**

一旦你接受这个原则，`session_search` 为什么必须存在、为什么必须先检索再摘要、为什么不能直接原文回放，就都顺了。

#### 为什么这点很值得你学

很多 Agent 项目做长期记忆时，最先想到的是“怎么存更多过去”。  
Hermes 这里更值得学的反而是：

**怎么让过去在需要时被拿回来，但不要长期压在主模型脑子里。**

这不是补功能，而是在做上下文卫生管理。  
长期 Agent 做到后面，往往不是“记不住”先出问题，而是“记得太多、带得太多、分不清当前和过去”先出问题。

`session_search` 正是在处理这个问题。
